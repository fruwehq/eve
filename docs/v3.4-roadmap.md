# v3.4 Roadmap — CI/runtime performance and Python consolidation

Companion to [v3.3-roadmap.md](v3.3-roadmap.md).

v3.3 proved Ubuntu host support and Docker runtime deployability, but
it also exposed two structural costs:

1. CI spends too much wall-clock time rebuilding and retesting the
   same Docker runtime.
2. The runtime image carries two application language stacks. Ruby
   is used for core orchestration and tests, while Python is used for
   the TUI and dependency management. The Docker image currently builds
   Ruby and Python from source, which is slow and expensive.

v3.4 makes Eve faster to verify and simpler to ship by reducing CI
duplication, moving container builds to prebuilt runtimes, and starting
the project-wide migration toward Python as the single application
language.

## Goals

1. **CI under 15 minutes on a clean PR run.** A PR should complete the
   required macOS, Ubuntu, and Docker slim checks in less than 15
   minutes excluding GitHub queue time.
2. **No source-built language runtimes in Docker.** The runtime image
   must use upstream/prebuilt language runtimes or distro packages.
   Building Ruby or Python from source in CI is out of scope for the
   normal PR path.
3. **No duplicate Docker runtime builds in required checks.** The slim
   image is built once per workflow run, then reused for smoke tests and
   size checks.
4. **Python becomes the target orchestration language.** New
   orchestration code should be Python. Existing Ruby orchestration is
   migrated behind compatibility-preserving command boundaries.
5. **Keep v3.3 deployability intact.** Ubuntu host support, Docker
   slim/full images, `EVE_HOME`, and TUI flows must keep working during
   the migration.

## Current Timing Baseline

From the v3.3 final PR runs:

| Job | Approximate time | Notes |
| --- | ---: | --- |
| `test (macos-latest)` | 7 minutes | Host suite only. |
| `docker-runtime-slim` | 12 minutes | Builds slim image, runs `make help`, measures compressed size. |
| `test (ubuntu-24.04)` | 18 minutes | Full host suite plus Linux-only Docker smoke. |
| `docker-runtime-full` | skipped on PR | Schedule-only/nightly. |

The slowest required PR leg is currently Ubuntu at about 18 minutes.
Repeated pushes during review multiply that cost.

## Known Cost Centers

| Area | Current cost | v3.4 direction |
| --- | --- | --- |
| Docker language build | Ruby 3.4 and Python 3.13 are compiled from source in `docker/Dockerfile` | Use a prebuilt Python base image and remove Ruby from the slim runtime once orchestration migrates |
| Duplicate Docker smoke | `docker-runtime-slim` builds/smokes the image, and Linux `make test` also runs `scripts/test-docker` | Build slim once and make the Ubuntu matrix consume the same artifact or skip only the duplicate Docker smoke when the dedicated Docker job is required |
| Docker cache | No persistent build cache across PR runs | Add BuildKit/buildx cache with GitHub Actions cache backend |
| Image size measurement | `docker save | gzip` adds roughly two minutes | Keep it in the dedicated Docker job only, not in host matrix jobs |
| Ruby test/runtime setup | `ruby/setup-ruby` and Bundler run on every host job | Remove after Ruby orchestration has been migrated |
| Mixed language ownership | Ruby SDK + scripts, Python TUI | Move shared SDK/config/state/plugin logic to Python modules used by both CLI and TUI |

## Implementation Order

### 1. Measure and publish CI timing

- Add a lightweight timing summary step to each required job.
- Record the slowest steps in a CI artifact or job summary.
- Extend `scripts/test-ci-pipeline` so it asserts the intended workflow
  shape: one dedicated slim image build, required macOS/Ubuntu host jobs,
  and schedule-only full image build.

Acceptance:

- PR job summaries show step-level durations for macOS, Ubuntu, and
  Docker slim.
- `make test.ci-pipeline` verifies the optimized workflow contract.

### 2. Add Docker layer caching

- Use `docker/setup-buildx-action` and BuildKit cache import/export for
  the slim runtime build.
- Scope cache keys by Dockerfile path, lockfiles, and architecture.
- Keep cache misses explicit. If cache restore fails, the job should
  continue as a normal rebuild, but the logs must say cache was missed.

Acceptance:

- First clean PR run still passes.
- A second PR run with no Dockerfile/dependency changes reuses cache and
  cuts slim image build time materially.
- `docker-runtime-slim` still reports compressed image size and enforces
  the v3.3 threshold.

### 3. Remove duplicate Docker work from the Ubuntu matrix

The v3.3 Linux `make test` path intentionally includes Docker smoke
coverage. v3.4 should keep the coverage but avoid building the same
image twice.

Preferred shape:

- `test (ubuntu-24.04)` runs host tests only.
- `docker-runtime-slim` is a required PR check and owns:
  - building `eve/eve:slim`
  - `docker run --rm eve/eve:slim make help`
  - compressed size measurement
  - any Linux-only Docker runtime smoke

If keeping Docker smoke inside `make test` is still desirable for local
Linux users, add an explicit environment variable such as
`EVE_SKIP_DOCKER_SMOKE=1` in the Ubuntu CI job with a logged message that
the dedicated required Docker job owns that coverage. Do not silently
skip.

Acceptance:

- Required PR checks still include `test (ubuntu-24.04)` and
  `docker-runtime-slim`.
- Ubuntu host CI no longer builds the slim image when the dedicated
  Docker job is present.
- Local Linux `make test` behavior remains documented.

### 4. Stop compiling Python in Docker

- Replace the source-built Python stage with a prebuilt Python runtime
  base image, preferably `python:3.13-slim-bookworm` or a newer official
  slim variant that supports the project.
- Install Python dependencies from the lockfile without dev/test
  dependencies in runtime images.
- Keep host CI on `actions/setup-python` for macOS and Ubuntu.

Acceptance:

- `docker/Dockerfile` contains no Python source download/build step.
- `docker-runtime-slim` still passes.
- TUI starts inside `eve/eve:slim`.

### 5. Inventory Ruby runtime dependencies

Produce `docs/v3.4-ruby-inventory.md` listing every Ruby entry point,
library dependency, and test dependency.

Minimum inventory:

- CLI scripts under `scripts/` with Ruby shebangs.
- `core/sdk/*.rb` modules and their public call sites.
- Ruby gems from `Gemfile`.
- Tests that execute Ruby scripts directly.
- Make targets that assume Ruby/Bundler.

Acceptance:

- Inventory exists with file-level ownership and migration priority.
- Each Ruby script is classified as:
  - keep until final compatibility cutover
  - port directly to Python
  - delete/merge into another command

### 6. Introduce Python SDK parity

- Create a Python SDK package for shared workdir, config, state,
  schema, plugin manifest, and command contract logic.
- Start with direct parity tests against the existing Ruby behavior.
- Keep command-line output contracts byte-for-byte compatible where
  other scripts, Make targets, or tests consume them.

Acceptance:

- Python SDK has tests for Workdir, config, state, and plugin manifest
  behavior.
- Ruby and Python implementations produce identical outputs for selected
  fixture inputs.
- No user-facing command behavior changes.

### 7. Port orchestration commands incrementally

Port Ruby entry points in dependency order. Start with low-risk leaf
commands, then move shared dispatch/state commands.

Suggested order:

1. Read-only commands: `catalog-options`, `plugin-list`,
   `instance-list`, `instance-paths`.
2. Config/state helpers: `config-env`, `config-save`, `instance-state`.
3. Provider/package dispatch helpers.
4. Instance create/delete/resolve/status/view.
5. Test helpers and doctor/check-required.

Each port must preserve CLI flags, stdout/stderr shape, exit codes, and
schema validation behavior.

Acceptance:

- Each migrated command has Python tests.
- Existing `make test` remains green after every migration commit.
- No wrapper silently falls back from Python to Ruby.

### 8. Remove Ruby from runtime images

After orchestration no longer depends on Ruby:

- Remove Ruby and Bundler from `docker/Dockerfile`.
- Remove Ruby setup from the runtime image build path.
- Remove Bundler install steps from Docker runtime stages.
- Keep Ruby in CI only if unreplaced developer/test tooling still needs
  it; otherwise remove from CI too.

Acceptance:

- `eve/eve:slim` has no `ruby` executable unless pulled in by an
  unrelated system package.
- `docker-runtime-slim` passes without Ruby.
- Image compressed size decreases or stays below the v3.3 threshold.

### 9. Revisit full runtime image

- Keep `eve/eve:full` schedule-only unless a PR explicitly changes local
  QEMU/Vagrant behavior.
- Apply the same prebuilt runtime and cache improvements to full.
- Measure whether `qemu-system-aarch64` belongs in full or should be a
  separate `eve/eve:local-qemu` variant.

Acceptance:

- Scheduled full image build remains green.
- Full image size and build time are recorded in acceptance evidence.

## Non-goals

- Removing macOS host CI. macOS remains a supported host until a later
  roadmap explicitly removes it.
- Replacing Terraform, Terramate, Vagrant, or QEMU.
- Changing provider behavior or cloud-provider credentials.
- Rewriting guest-side provisioning languages. Linux guest provisioning
  stays bash; Windows guest provisioning stays PowerShell.
- Hiding missing tools behind fallbacks. Missing tools must still fail
  clearly or be reported by `make doctor`.

## Acceptance

- A clean PR run completes required checks in less than 15 minutes,
  excluding GitHub queue time.
- Required PR checks are still:
  - `test (macos-latest)`
  - `test (ubuntu-24.04)`
  - `docker-runtime-slim`
- `docker-runtime-slim` builds once, smokes once, reports compressed
  size, and enforces the size threshold.
- `docker/Dockerfile` no longer compiles Ruby or Python from source.
- Slim runtime no longer contains Ruby after the Python orchestration
  migration is complete.
- `make test` is green on macOS and Ubuntu.
- `make test.ci-pipeline` verifies the optimized workflow contract.
