# v4.5 — FSM platform + settings-as-context (design)

Spine doc for roadmap bugs **N** (central FSM platform) and **O** (settings are
FSM context). Engine lives in core; **every machine is declared in YAML by the
plugin that owns it**. The interface is kept thin so the engine can later be
replaced by a cns-based implementation (ref: github.com/fruwe/cns_statemachine).

## Why

Today state is scattered (`eve_sdk/state.py` enums, `state_machine.py` derivation,
`desired_state_for`/`provider_state_for`/`record_*` across dispatch) and **config
lives outside it entirely**. So `providers-status` could resolve config one way
and dispatch another — Vultr ✗ while creatable, AWS "configured" off ambient
`~/.aws`. The cure: one engine that owns both **state** and **context (settings)**
and hands them to every command identically. If config is always read from the
machine's context, status and dispatch *physically cannot* diverge.

## Core model

- **Machine definition** — declarative YAML a plugin ships. Describes one *kind*
  of machine (a provider, a package, …): its states, transitions, and the shape
  of its context (variables + settings).
- **Machine instance** — a live entity the engine tracks: `(definition, key,
  current_state, context)`. Keys: a provider id; an instance name; an
  `(instance, package)` pair; etc.
- **Context** — the machine's state variables, including **resolved settings**.
  The single source commands read config from.
- **Event** — a verb that may cause a transition. Events are the existing CLI
  verbs (`configure`, `up`, `down`, `provision`, `install`, `probe`, …).
- **Transition** — `from --on(event)--> to`, optionally gated by a **guard** over
  context, effected by an **action** (a plugin command). Action succeeds → land
  in `to`; fails → an error state.
- **Observe/reconcile** — a read-only command maps real-world status back into a
  state (e.g. connectivity → `reachable`, terraform status → `running`). Keeps
  the machine honest about drift.
- **Storage** — per machine kind: `host` (persisted state record) or `client`
  (derived live, not persisted). The engine owns persistence; machines don't.

## Plugin-declared YAML (the schema)

A `state_machine:` block in `eve-plugin.yaml` (or a referenced `fsm.yaml`):

```yaml
state_machine:
  scope: provider                 # provider | instance | package | os | bundle
  initial: unconfigured
  storage: host                   # host | client

  # Context = state variables. `setting`-sourced vars are resolved by the engine
  # (default -> global override -> per-instance override) and exposed to every
  # command of this machine via env_var. `null` default == the explicit "unset".
  context:
    api_key:
      source: secret              # setting | secret | observed | derived
      type: secret
      default: null               # unset
      scope: global               # global | instance | both
      env_var: VULTR_API_KEY
    region:
      source: setting
      type: string
      default: ""
      scope: both
      env_var: VULTR_REGION

  states:
    unconfigured: {}
    configured: {}
    reachable: {}

  transitions:
    - { from: "*",          on: resolve, to: configured,   guard: "api_key != null" }
    - { from: configured,   on: probe,   to: reachable,    action: {command: connectivity}, guard_on_result: "reachable == 'yes'" }
    - { from: "*",          on: resolve, to: unconfigured, guard: "api_key == null" }

  observe:                        # read-only reconciliation
    command: connectivity         # emits {configured, reachable, notes}
    maps: { reachable: reachable, configured: configured }
```

Notes:
- **Guards** are simple expressions over context (no arbitrary code); a small,
  safe evaluator. Result-guards (`guard_on_result`) check an action's JSON output.
- **Actions/commands** are the plugin's existing `commands:` — the FSM references
  them by name, it doesn't introduce a second command surface.
- A package machine looks the same with `scope: package`, states
  `{missing, installing, installed, failed, removed}`, events
  `{install, status, down}`, and context for its settings.

## Repository split (generic engine vs eve adapters)

The engine is its **own repository** — a generic, eve-agnostic FSM library; eve
depends on it one-way (like every other agnostic-core boundary). This makes the
"swap for a cns-based engine later" goal enforceable: the repo is the swappable
unit with a published interface, not a module that can drift into eve internals.

| Lives in the **engine repo** (generic) | Lives in **eve** (adapters) |
|---|---|
| Parse/validate a machine definition (YAML) | The `state_machine:` YAML in each plugin |
| Hold `(state, context)`; evaluate guards; fire transitions | Resolve settings → context (default→global→instance); inject secrets |
| Determine valid events; expose current state | Map eve commands ↔ FSM actions; entity→machine wiring |
| **Interfaces** for storage, action execution, context provider | **Implement** those interfaces (Workdir storage, run plugin command, ConfigEnv/Secrets) |

The engine never imports eve and never names a provider/package/OS — it just runs
whatever machine YAML + callbacks it is handed.

## The platform interface (the engine↔eve boundary)

Thin, storage- and engine-agnostic — this *is* the repo boundary and the cns seam:

```python
class Machine:                    # one definition + key
    state: str
    context: Mapping[str, str]    # resolved settings + observed vars

class FSM:
    def load(kind, plugin) -> Definition          # parse + validate YAML
    def machine(scope, key, *, registry=None) -> Machine
    def context(scope, key) -> Mapping[str, str]  # resolved settings (THE helper M/O need)
    def fire(scope, key, event, *, extra_args=()) -> Result   # run transition (guard->action->persist)
    def observe(scope, key) -> Machine            # reconcile from the observe command
    def states(scope, key) -> StateView           # current state + available events (for TUI/CLI)
```

`context()` is the one shared resolver from bug M: `providers-status`, dispatch,
and connectivity all call it, so they cannot see different config/secrets.

## Settings-as-context (the bug-killer)

Resolution order, done once by the engine: **plugin default → global override
(`~/.config/eve/config.yaml` / `eve settings ...` / TUI) → per-instance override
(instance entry / `eve instance settings ...` / TUI)**. The resolved value is put
into context and exposed to every command of that machine via `env_var`. There is
no other path to config — no plugin reads `~/.aws` or ambient env behind eve's
back; if a provider needs ambient creds, it declares them as a `setting` eve
resolves and injects.

## Mapping current eve → machines

| Machine (scope) | States (today → YAML) | Owner plugin |
|---|---|---|
| provider | unconfigured/configured/reachable (new; was the secret-less probe) | provider |
| instance | absent/creating/running/stopped/error (`PROVIDER_STATES`) + desired | provider |
| provision | unknown/provisioning/provisioned/error (`PROVISION_STATES`) | provider (os bootstrap) |
| package | missing/installing/installed/failed/removed (`PACKAGE_STATES`) | package |
| os | (image/bootstrap readiness) | os |
| bundle | derived from member packages | bundle |

`state.py`'s hardcoded enums move into the owning plugins' YAML; `state_machine.py`
+ the `*_state_for`/`record_*` functions collapse into `FSM`. `Engine`/`Workdir`
keep persistence, but behind `FSM`'s storage seam.

## Storage (host vs client)

- **host** — durable per-entity state record (instances, provision, packages).
  Lives under `~/.local/state/eve/` (roadmap P).
- **client** — derived live, not persisted (provider configured/reachable is
  recomputed via `observe`; cached briefly only).

## Open decisions (need your call)

1. **Instance lifecycle ownership** — does the *provider* plugin declare the
   instance FSM (lifecycles differ: vultr vs docker vs local-qemu), or is there a
   shared core instance FSM with provider-supplied transition actions? I lean
   **provider-declared** for honesty; it's more YAML per provider.
2. **Guard expression language** — minimal safe DSL (`==`, `!=`, `&&`, presence)
   vs. something richer. I lean **minimal**.
3. **Bundle/OS machines** — model now, or stub their states until B/§14 settle?
   I lean **stub** (define scope, minimal states) to keep N/O focused.
4. **One YAML block vs separate `fsm.yaml`** — inline in `eve-plugin.yaml` keeps
   it together; a separate file keeps manifests small. I lean **inline** with an
   optional `$ref`. (Orthogonal to the engine repo split — plugins declare their
   machine YAML in eve either way; the engine just parses it.)
5. **Engine repo home + name** — a generic FSM library, eve-agnostic. Under the
   eve org (`fruwehq/…`) or your personal org alongside cns (`fruwe/…`)? Generic
   name (`statemachine`, a modern cns) vs eve-scoped (`eve-fsm`)? I lean a
   **generic name** since the whole point is reuse + a future cns-based swap, but
   it's your call where it lives.
