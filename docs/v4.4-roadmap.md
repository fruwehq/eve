# v4.4 Roadmap — A plugin-agnostic core (zero provider *or* package knowledge)

Status: planned (2026-06-23). v4.0–v4.3 built the agnostic plugin host and
finalized the CLI/TUI. v4.4 finishes the job the host was meant to enable:
**the `eve` core knows nothing about any provider, any package, or the
functionality they offer.** It only defines the plugin interface and the
extension contract. No provider id (`aws`, `gcp`, `truenas`, `raspberry-pi`,
`vultr`, `local-qemu`, `docker`) and no package id (`sunshine`, `nomachine`,
`rdp`, `rustdesk`, `splashtop`, `thinlinc`, `vnc`, `xpra`, `waypipe`, …) — nor
any provider/package-specific field — should appear in core code, schemas,
defaults, or the TUI. They live only in plugin manifests and in test fixtures.

## Philosophy — core is the host, not a provider/package catalog

The core repo (`eve_sdk/`, `scripts/`, `tui/`, `config/`, `core/schema/`) is
the **host**: it discovers plugins, aggregates their contributed catalog, and
dispatches to their commands. It must not name, enumerate, special-case,
validate, or launch any concrete provider or package. Today both provider ids
**and** package ids leak into core — in config defaults/schemas, in `config.py`
env-var mappings, in remote-client launchers, and in dispatch/profile logic.
v4.4 removes every leak so a new provider or package can be added purely by
authoring a plugin, with **zero core edits**.

## Why now — what's still broken

With every plugin source removed, the TUI still listed providers, and core
still "knew" about packages. A re-grep of core production code
(`eve_sdk/`, `scripts/`, `config/`, `core/schema/`, `tui/`, excluding
tests/fixtures/docs) shows the remaining leaks:

**Provider knowledge (~24 hits):**
- `config/defaults.yaml` + `core/schema/config.schema.json`: `aws`, `gcp`,
  `truenas`, `raspberry-pi` sections.
- `core/schema/resolved-instance.schema.json`: per-provider OS-image fields
  `aws_ssm_parameter`, `gcp_image_project`, `gcp_image_family`, `vultr_os_id`.
- `core/schema/plugin-manifest.schema.json`: the same provider catalog OS
  fields + the closed `engines` enum.
- `tui/settings.py`: `aws`/`gcp`/`truenas`/`raspberry-pi` section labels +
  `raspberry-pi.*` field metadata.
- `stacks/{aws,truenas,vultr}`: legacy in-repo provider implementations.

**Package knowledge (~40 hits — the originally-missed half):**
- `config/defaults.yaml` + `core/schema/config.schema.json`: `moonlight`,
  `nomachine`, `rdp`, `rustdesk`, `splashtop`, `sunshine`, `thinlinc` sections.
- `eve_sdk/config.py` `MAPPINGS`: every package env-var row
  (`SUNSHINE_*`, `THINLINC_*`, `SPLASHTOP_*`, `RUSTDESK_SERVER`,
  `NOMACHINE_VERSION`, `RDP_GATE_USER`). The provider-agnostic emission only
  scans providers (`load_all("provider")` at `config.py:116`), so packages are
  still hardcoded.
- `eve_sdk/remote_launch.py`: core builds the `sunshine`/`moonlight`/
  `rustdesk`/`waypipe`/`xpra` **remote-client launchers** itself.
- `eve_sdk/profile_resolve.py`: branches on `"sunshine"`, `"vnc"`, `"xpra"`,
  `"rdp"`, `"thinlinc"` in the bundle.
- `eve_sdk/dispatch.py:275`: branches on `package_id == "rustdesk"`.
- `tui/settings.py`: `moonlight`/`sunshine`/`rdp`/`rustdesk`/`thinlinc` labels.

Two structural causes compound the leaks:
1. **Discovery isn't source-aware.** `plugin_paths()` walks `.eve/plugins/`
   regardless of the configured source set, so materialized plugins linger
   after a source is removed (the immediate cause of the lingering list).
2. **The prune shipped in v4.4.0 was symlink-only**, so a plugin materialized
   as a real directory survived a source removal. (Fixed during this pass —
   see "Achieved".)

## Goals

1. **Zero provider/package/engine literals in core.** No provider or package
   id — and no provider/package-specific field — appears in `eve_sdk/`,
   `scripts/`, `config/`, `core/schema/`, or the TUI outside the plugin-
   interface contract.
2. **Source-aware discovery + full reconciliation.** The installed plugin set
   is exactly what the configured sources materialize — never more — and the
   synced root is fully reconciled (symlinks *and* real dirs).
3. **Plugin-owned config & schemas.** A provider's *and* package's config
   fields, defaults, env vars, and validation live entirely in their manifest,
   surfaced dynamically by core.
4. **Plugin-owned remote-client launchers.** How to launch a remote client is
   declared by the package that owns it, not hardcoded in core.

## Achieved in this milestone (v4.4.0)

Status: §1, §2, §3, §4, §5, §6, §7, §10, §12, §13 done and verified; §11 scope
widening done (provider/OS-id ban across `scripts/`, `eve_sdk/`, `tui/`,
`config/`, `core/schema/` — core is now provider/OS-id free). **Remaining:
§8 (package-owned remote launchers — the deep one) and §9 (legacy stacks,
blocked on re-pointing `test-terraform` at the external stacks); §11's
package-id ban waits on §8.**

- **Provider env vars are provider-owned.** Dropped the `raspberry_pi` rows
  from `ConfigEnv.MAPPINGS` (now `env_var`s on the `raspberry-pi` manifest)
  and replaced the hardcoded `PATH_ENV_NAMES` provider set with per-row
  `is_path`. Core no longer declares a single provider env var, so none emit
  when no provider is installed. Config section renamed `raspberry_pi` →
  `raspberry-pi` to match the provider id. (§1 provider half + earlier work.)
- **Package config is package-owned (§2 + §3).** `_plugin_mappings()` scans
  package plugins as well as providers; the package rows (`SUNSHINE_*`,
  `THINLINC_*`, `SPLASHTOP_*`, `RUSTDESK_SERVER`, `NOMACHINE_VERSION`,
  `RDP_GATE_USER`) are deleted from `MAPPINGS` and declared on each package
  manifest's `config_schema` (eve-packages-linux). `MAPPINGS` now holds only
  true-core rows (`global`, `display`, `vagrant`, `moonlight`).
- **Core config layer is provider/package-free (§1).** `config/defaults.yaml`
  and `core/schema/config.schema.json` carry only core sections (`global`,
  `human_user`, `windows`, `vagrant`, `display`, `moonlight`).
- **Core schemas name no provider (§4, §5).** `resolved-instance.schema.json`
  drops the per-provider OS-image fields and location keys (freeform via
  `additionalProperties`); `plugin-manifest.schema.json` drops the provider
  catalog OS fields. The engines enum stays (it is the extension contract).
- **TUI is dynamic (§6).** `CONFIG_SECTIONS` lists only core sections; provider
  and package sections render from installed plugins (display_name → label,
  manifest → descriptions). Provider/package `FIELD_META` removed.
- **Source-aware discovery (§7).** `plugin_paths()` only descends into synced-
  root exposures backed by a configured source — the durable fix for the
  lingering Providers list (a stale materialization / empty source set yields
  nothing). Verified hermetic fixtures still bypass this.
- **TUI provider list comes only from plugins**, and the **TUI settings
  provider section set is dynamic** (derived from installed providers, not a
  hardcoded `{aws,gcp,truenas}`).
- **Startup Cancel-button flash fixed** (tracks a real subprocess, not the
  in-process refresh; `display:none` until `#root.busy`) and a **refetch
  indicator** shows on the summary bar while caches reload.
- **Full synced-root reconciliation.** ⚠️ Behavior change from the first cut:
  `prune_plugins()` now removes *any* entry in the eve-managed synced root
  (`.eve/plugins/`) that is not backed by a configured source — **symlink or
  real directory** — not just symlinks. The synced root is eve-owned (entirely
  materialized by `sync()`), so force-pruning it recovers from stale/dirty
  materializations and forces a fresh re-download on the next pull. The safety
  boundary is **scope, not file type**: user-owned roots (`EVE_PLUGIN_ROOTS`)
  are a separate root, never passed to the pruner. (`sync()` calls it after
  materializing; the TUI calls it on plugin-screen close.)

> Note: the full prune alone does **not** empty the Providers list for a user
> whose `.eve/plugins/` is already dirty — `sync()`/the TUI must run once to
> reconcile. The durable fix is source-aware discovery (§7), which makes the
> on-disk root and the configured set unable to drift.

## Work items — make core fully agnostic

### 1. Config layer (defaults + schema) — keep only genuinely-core sections
Provider **and** package config sections live in core today. Classification
(done by checking which sections appear in a plugin manifest's
`config_schema`/`env:` vs. only in core):

| Section | Owner | Evidence |
|---|---|---|
| `global`, `human_user`, `windows`, `vagrant` | **core** | consumed by core (`global` everywhere; `human_user` in `package_dispatch`; `windows` is the OS family + `scripts/provision`; `vagrant` is an engine in `vagrant_dispatch`) |
| `display` | **core (shared)** | consumed by several packages (`sunshine`, `rustdesk`, `vnc`) — a cross-cutting VM display setting, not owned by any one package |
| `moonlight` | **sunshine's client (package-owned)** | moonlight is the **client** paired with the `sunshine` streaming **host/server** (sunshine's manifest declares the `pair-moonlight` action + a `moonlight` env entry). It is not core; its config rows (`config.py`) and launcher (`remote_launch.py`) move with the sunshine package (§8) |
| `aws`, `gcp`, `truenas`, `raspberry-pi` | **provider** | provider manifests |
| `nomachine`, `sunshine`, `rdp`, `rustdesk`, `splashtop`, `thinlinc` | **package** | `kind: package` manifests in `eve-packages-*` |

**Do:** remove the provider rows (`aws`, `gcp`, `truenas`, `raspberry-pi`) and
the package rows (`nomachine`, `sunshine`, `rdp`, `rustdesk`, `splashtop`,
`thinlinc`, and `moonlight` — sunshine's client) from `config/defaults.yaml`
and `core/schema/config.schema.json` (incl. the `nullable_hdmi_connector`
def). Keep only `global`, `human_user`, `windows`, `vagrant`, `display`
(`moonlight`'s fields move into sunshine's manifest via §2/§3; its launcher
via §8). Provider/package fields, defaults, and validation already come from
each manifest's `config_schema`; `check-required` reads "required" markers
from manifests.

### 2. Package-owned config emission (the missing core mechanism)
`_provider_mappings()` in `eve_sdk/config.py` only scans providers
(`load_all("provider")`). **Do:** generalize and rename it (e.g.
`_plugin_mappings`) to scan package plugins too (`load_all("provider")` ∪
`load_all("package")`), so package config fields are emitted from their
manifests exactly like provider fields. After this, `MAPPINGS` keeps only
true-core rows (`global`, `human_user`, `windows`, `vagrant`, `display`) and
the package rows (`SUNSHINE_*`, `MOONLIGHT_*` [sunshine's client], `THINLINC_*`,
`SPLASHTOP_*`, `RUSTDESK_SERVER`, `NOMACHINE_VERSION`, `RDP_GATE_USER`) are
deleted (declared by their package manifests instead).

### 3. `eve-packages-*` counterpart (release-coupled)
Package manifests carry only an `env:` allowlist today — they declare **0**
`config_schema` `env_var`s (verified: `grep -c env_var` is 0 for sunshine,
moonlight, thinlinc, splashtop, rustdesk, nomachine, rdp). **Do:** add
`config_schema` `env_var` declarations to each package manifest in
`eve-packages-linux` and `eve-packages-windows`, mirroring what
eve-providers#7 did for `raspberry-pi`. The `moonlight` client fields land in
the **sunshine** manifest (sunshine = host/server, moonlight = its client).
⚠️ **Release-coupled**: merging core (§2) before these package PRs would
silently drop every package env var.

### 4. Resolved-instance schema — drop provider-specific fields
`core/schema/resolved-instance.schema.json` hardcodes per-provider OS-image
fields `aws_ssm_parameter`, `gcp_image_project`, `gcp_image_family`,
`vultr_os_id`, and the `location` object hardcodes
`aws`/`gcp`/`vultr`/`local-qemu`/`raspberry-pi`/`truenas` keys. **Do:** move
all provider-specific OS-image metadata under `additionalProperties` (freeform
per-provider catalog data), and drop the per-provider `location` keys (keep
`name` + a generic `provider_location` `$def` via `additionalProperties: true`,
which the object already allows).

### 5. Plugin-manifest schema — drop provider catalog fields + engine enum
`core/schema/plugin-manifest.schema.json` hardcodes the same provider catalog
OS fields (`gcp_image_project`, `gcp_image_family`, `vultr_os_id`) and pins
`engines: [terraform, vagrant, qemu, metal, docker]`. **Do:** open the catalog
OS field set (`additionalProperties` for per-provider image metadata). **Decide
on the engine enum:** keep it as the documented extension contract (core may —
it *is* the interface), or open it and validate lazily at dispatch. Lean: keep
as interface contract; ensure no provider-id-shaped value leaks in.

### 6. TUI settings — dynamic sections + labels, incl. packages
`tui/settings.py` `CONFIG_SECTIONS` still hardcodes provider **and** package
labels (`aws`, `gcp`, `truenas`, `raspberry-pi`, `moonlight`, `sunshine`,
`rdp`, `rustdesk`, `thinlinc`), and `FIELD_META` carries `raspberry-pi.*`.
**Do:** render provider *and* package sections from installed plugins (id +
manifest `display_name`); descriptions/examples come from each manifest. Core
keeps only the genuinely-core section labels (§1's table).

### 7. Source-aware discovery (the durable fix for the lingering list)
`PluginManifest.plugin_paths()` walks `.eve/plugins/` for any subdir, so stale
materialized plugins show after a source is removed. The shipped full prune
(see Achieved) reconciles the synced root when `sync()`/the TUI runs, but the
durable fix is to make discovery honor the configured source set for synced
roots, so the on-disk root and the configured set cannot drift.
`EVE_PLUGIN_ROOTS_EXCLUSIVE` hermetic test fixtures continue to bypass this
(they are not source-managed).

### 8. Remote-client launchers + package dispatch — package-owned  *(remaining — the deep item)*
`eve_sdk/remote_launch.py` hardcodes the `sunshine`, `moonlight`, `rustdesk`,
`waypipe`, `xpra` remote-client launchers; `eve_sdk/profile_resolve.py`
branches on `"sunshine"`/`"vnc"`/`"xpra"`/`"rdp"`/`"thinlinc"`; and
`eve_sdk/dispatch.py:275` branches on `package_id == "rustdesk"`. This is core
knowing package functionality. **Do:** let each package *declare* its
remote-client launcher (command template) and its profile/dispatch hooks via
the manifest, and have core invoke them generically. `moonlight` is not a
standalone package — it is `sunshine`'s **client** (sunshine = host/server,
moonlight = client, per sunshine's `pair-moonlight` action) — so the
`remote-moonlight` launcher and the `moonlight` config rows move **with the
sunshine package** (or a shared `_clients` package that sunshine owns), not
into core and not as an independent decision. This retires the last
package-id references in `config.py`'s `moonlight` rows alongside the launcher.

### 9. Remove legacy provider stacks  *(remaining — blocked on test-terraform rework)*
`stacks/{aws,truenas,vultr}` (+ their `.terraform.lock.hcl`) are legacy
in-repo provider implementations (no `gcp` stack exists on disk). Per the
provider-plugin boundary, provider infra lives in the provider plugins.
**Do:** delete them; keep only `stacks/config.tm.hcl` (legacy shared globals)
until that too is retired.

### 10. Comments & examples
No provider/package id should appear even as an illustration outside plugin
manifests and test fixtures. v4.4.0 already cleaned `eve_sdk/config.py`,
`eve_sdk/profile_resolve.py`, `tui/widgets.py`, and
`config/catalog.local.example.yaml`; finish any remaining example references.

### 11. Extend core-boundary lint to ban provider *and* package ids
`scripts/test-core-boundary` derives `all_ids()` from
`PluginManifest.load_all("provider")` + catalog `oses` only — **package ids
are not in the banned set**, so the package leaks above never failed CI.
**Do:** (a) add package ids (`load_all("package")`) to the banned set, and
(b) extend the scan scope to `core/schema/`, `config/`, and `tui/`. The
context-aware `COMMON_TOKEN_IDS` matching already handles dual-meaning names
(e.g. `docker` the CLI vs the id); reuse it for `steam`/`discord`/`vnc` if
needed. This lands **after** the leaks are removed, otherwise CI fails on
the current references.

## Parallel track — source-resolution ergonomics (independent of agnosticism)

§12–§13 improve how a source is *specified and exposed*. They are independent
of the agnosticism items (§1–§11) and can land in parallel with them. A source
is `(repo, branch, folder)` with `folder` defaulting to `/` (repo root).

### 12. TUI: add Branch + Folder to add-source
The registry layer already supports `(url, subdir, ref)` and `parse_sources`
already validates `subdir` (strips slashes, rejects `..` via
`_is_unsafe_subdir`). The gap is purely the TUI:

- `tui/plugins.py:62` `add_url(url, *, ref="", source_id="", auth="none")`
  has **no `subdir` param** — entries are always created with `subdir=""`.
- `tui/widgets.py` `_open_add_url` only parses `url#ref`; there is no folder
  field, and the sources table shows no folder.

**Do:** plumb `subdir` through `add_url` (and through recommended-source
entries where a subdir applies), and give the add-source prompt explicit
**Branch** and **Folder** inputs (Folder defaulting to `/`). Reuse
`parse_sources`' existing validation so the TUI cannot create an unsafe
subdir — do **not** re-implement the `..` check. Show `subdir` as a column in
the sources table (`_reload` in `widgets.py`) so the user can see which folder
each source points at. Add a TUI/registry test covering add-with-folder round-
tripping into `.eve/plugin-sources.yaml`.

### 13. `.eve/` folder convention in source exposure
Today a source's folder is exposed verbatim as the plugin root. Convention: a
source is `(repo, branch, folder)` with `folder` default `/`; after the folder
is resolved, **if it contains a `.eve/` subdirectory, expose that as the
plugin root; otherwise expose the folder itself.** This lets users tuck eve's
files out of their repo's top level without pointing the source at a deeper
folder by hand.

**Do:** in the sync/exposure path (`_expose_subdir` / `sync` in
`eve_sdk/registry.py`), after sparse-checking-out the source's `subdir`,
check for a `.eve/` subdirectory in the resolved folder and expose that when
present. Document the precedence (explicit `.eve/` wins over the folder root)
and that it is transparent — the user still specifies only the folder. Add
tests for both shapes: a source folder with `.eve/` (uses it) and without
(uses the folder). This builds on §12 (which lets the user specify the
folder) and is the concrete realization of the `.eve/` convention noted in
the v4.1 ideas memo.

## Non-goals

- Changing the plugin manifest format or the discovery interface beyond what
  agnosticism requires.
- Removing the engine/kind concept — that *is* the extension contract.
- Removing provider/package ids from test fixtures or example plugin manifests.
