# v4.4 Roadmap — A provider-agnostic core (zero provider/package knowledge)

Status: planned (2026-06-23). v4.0–v4.3 built the agnostic plugin host and
finalized the CLI/TUI. v4.4 finishes the job the host was meant to enable:
**the `eve` core knows nothing about any provider, package, or the
functionality they offer.** It only defines the plugin interface and the
extension contract. No provider id (`aws`, `gcp`, `truenas`, `raspberry-pi`,
`vultr`, `local-qemu`, `docker`) and no provider-specific field should appear
in core code, schemas, defaults, or the TUI — only in plugins (and in test
fixtures as data).

## Philosophy — core is the host, not a provider catalog

The core repo (`eve_sdk/`, `scripts/`, `tui/`, `config/`, `core/schema/`) is
the **host**: it discovers plugins, aggregates their contributed catalog, and
dispatches to their commands. It must not name, enumerate, special-case, or
validate any concrete provider or package. Today a handful of provider ids and
provider-specific config still leak into core; v4.4 removes every leak so a
new provider or package can be added purely by authoring a plugin, with **zero
core edits**.

## Why now — what's still broken

With every plugin source removed, the TUI still listed providers. Two causes:

1. **Discovery isn't source-aware.** `PluginManifest.plugin_paths()` walks
   `.eve/plugins/` regardless of the configured source set, so materialized
   plugins linger after their source is removed. (`eve pull` now prunes them
   on sync, and the TUI prunes on plugin-screen close — both shipped in this
   milestone — but a stale dir still shows until it's reconciled.)
2. **Core still names providers.** `config/defaults.yaml`,
   `core/schema/config.schema.json`, `core/schema/resolved-instance.schema.json`,
   and the TUI settings map hardcode `aws`/`gcp`/`truenas`/`raspberry-pi` — so
   even an empty core "knows" these names.

## Goals

1. **Zero provider/package/engine literals in core.** No provider id appears
   in `eve_sdk/`, `scripts/`, `config/`, `core/schema/`, or the TUI outside
   the plugin-interface contract.
2. **Source-aware discovery.** The installed plugin set is exactly what the
   configured sources materialize — never more.
3. **Provider-owned config & schemas.** A provider's config fields, defaults,
   env vars, and validation live entirely in its manifest, surfaced dynamically
   by core.
4. **No legacy provider implementations in core.** `stacks/{aws,gcp,vultr,truenas}`
   are gone; provider infra lives in the provider plugins.

## Achieved in this milestone (v4.4.0)

- **Plugin env vars are provider-owned.** Dropped the `raspberry_pi` rows from
  `ConfigEnv.MAPPINGS` (they are now `env_var`s on the `raspberry-pi` manifest)
  and replaced the hardcoded `PATH_ENV_NAMES` provider set with per-row
  `is_path`. Core no longer declares a single provider env var, so **none emit
  when no provider is installed.** Config section renamed `raspberry_pi` →
  `raspberry-pi` to match the provider id (the invariant `aws`/`gcp`/`truenas`
  already follow).
- **Orphan plugins are pruned.** `sync()` prunes materialized exposures whose
  source was removed; the standalone `prune_plugins()` lets the TUI drop them
  locally on plugin-screen close.
- **TUI provider list comes only from plugins.** `provider_pane_data()` reads
  the warm engine's provider plugin set; with none installed the Providers
  pane is empty.
- **TUI settings provider set is dynamic.** `SettingsScreen` derives its
  provider section set from installed providers instead of a hardcoded
  `{aws, gcp, truenas}`.
- **Startup Cancel-button flash fixed** — the Cancel button + expanded-output
  layout now track a real subprocess (`current_process`), not the in-process
  refresh, and the button is `display:none` until `#root.busy`.
- **Refetch indicator** — a `⟳ Refetching …` indicator shows on the summary
  bar while caches are invalidated and re-read (startup, plugin reload).

## Work items — make core fully agnostic

### 1. Config layer (defaults + schema) — remove provider sections
Provider config sections live in core today:

- `config/defaults.yaml`: `aws`, `gcp`, `raspberry-pi`, `truenas` (all empty —
  no defaults).
- `core/schema/config.schema.json`: the same sections, their field schemas,
  and the `nullable_hdmi_connector` definition (raspberry-pi-specific).

**Do:** drop provider sections from both; keep only core sections (`global`,
`display`, `moonlight`, `nomachine`, `sunshine`, `rdp`, `rustdesk`, `splashtop`,
`thinlinc`, `vagrant`, `human_user`, `windows`). Provider config fields,
defaults, and env vars already come from each manifest's `config_schema` (the
source of `config-env` emission and the settings UI). `check-required` then
reads provider "required" markers from manifests instead of `defaults.yaml`.

### 2. Resolved-instance schema — drop hardcoded location keys
`core/schema/resolved-instance.schema.json` `location` hardcodes
`aws`/`gcp`/`vultr`/`local-qemu`/`raspberry-pi`/`truenas` keys. The object
already allows `additionalProperties: true`. **Do:** drop the per-provider
keys; keep `name` + the generic `provider_location` `$def` referenced through
`additionalProperties` so any provider's location data validates.

### 3. Plugin-manifest engine/kind enums — open or mark as the contract
`core/schema/plugin-manifest.schema.json` pins
`engines: [terraform, vagrant, qemu, metal, docker]`. A closed enum means core
knows the engine set. **Decide:** either treat the enum as the documented
extension contract (core may keep it — it *is* the interface, not a provider),
or open it and validate engines lazily at dispatch. Lean: keep as the
interface contract; just ensure no provider-id-shaped value leaks in.

### 4. TUI settings — dynamic provider sections + labels
`tui/settings.py` `CONFIG_SECTIONS` still hardcodes `aws`/`gcp`/`truenas`/
`raspberry-pi` labels, and `FIELD_META` carries `raspberry-pi.*` entries.
**Do:** render provider sections from installed providers (id + manifest
`display_name`); provider field descriptions/examples come from the manifest.
Core keeps only the core section labels.

### 5. Source-aware discovery
`PluginManifest.plugin_paths()` walks `.eve/plugins/` for any subdir, so stale
materialized plugins show after a source is removed (the immediate cause of
the lingering Providers list). **Do:** make discovery honor the configured
source set for the synced roots (`.eve/plugins`), so the installed set is
exactly the configured set. `EVE_PLUGIN_ROOTS_EXCLUSIVE` hermetic test
fixtures continue to bypass this (they are not source-managed).

### 6. Remove legacy provider stacks
`stacks/{aws,gcp,vultr,truenas}` (+ their `.terraform.lock.hcl`) are legacy
in-repo provider implementations. Per the provider-plugin boundary (see
`AGENTS.md`), provider infra lives in the provider plugins. **Do:** delete
them; keep only `stacks/config.tm.hcl` (legacy shared globals) until that too
is retired.

### 7. Comments & examples
Reword the remaining provider-as-example references in core to be
provider-agnostic (the v4.4.0 commit already cleaned `eve_sdk/config.py`,
`eve_sdk/profile_resolve.py`, `tui/widgets.py`, and
`config/catalog.local.example.yaml`). No provider id should appear even as an
illustration outside plugin manifests and test fixtures.

### 8. Tests, fixtures & lint scope
Tests legitimately use provider ids as data (`mock-cloud`, `vultr`, …). Keep
test fixtures, but ensure no **core production code** path depends on a
concrete id. The core-boundary lint already enforces this for `scripts/` and
`eve_sdk/`; **extend its scan scope** to `core/schema/`, `config/`, and `tui/`
so a leaked provider id fails CI, not just the existing directories.

## Non-goals

- Changing the plugin manifest format or the discovery interface beyond what
  agnosticism requires.
- Removing the engine/kind concept — that *is* the extension contract.
- Removing provider ids from test fixtures or example plugin manifests.

## Sequencing

1. §5 source-aware discovery — directly fixes the lingering Providers list.
2. §1–§2 config + resolved-instance schema decoupling.
3. §4 TUI dynamic provider sections.
4. §6 remove legacy stacks.
5. §3 / §7 / §8 contract clarification, remaining comments, lint scope.
