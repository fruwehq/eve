# v3.1 Plan: Provider Boundaries and TUI Control Surface

## Summary

v3 made instances, provider plugins, package plugins, and state management real.
v3.1 should finish the separation: provider-specific implementation belongs in
provider plugins, package-specific implementation belongs in package plugins, and
core should stay a provider-agnostic orchestrator.

This also affects the TUI. The TUI should show provider health and expose
provider/package actions without hardcoding AWS, GCP, TrueNAS, Vultr, QEMU,
Raspberry Pi, Windows, or Ubuntu behavior.

## Design Direction

- Core owns:
  - catalog loading and instance resolution
  - plugin discovery and manifest validation
  - instance state files and operation history
  - shared support helpers such as per-instance workdir/state path creation
  - generic dispatch to provider/package commands
  - TUI rendering and command streaming
- Provider plugins own:
  - provider lifecycle implementation: `resolve`, `init`, `plan`, `up`, `down`,
    `start`, `stop`, `status`, `ip`, `ssh`
  - provider-specific Terraform/Terramate/Vagrant/metal files
  - provider-specific credential checks and login/admin actions where possible
  - provider-specific password retrieval behavior
- Package plugins own:
  - package install/status/down/reinstall implementation
  - package-specific host-side actions shown in the TUI
  - package-specific upload/config files when needed

`init` and `plan` are already part of the provider plugin command contract.
The remaining problem is implementation placement: today the manifests point
through `scripts/provider-plugin`, which then routes into central provider
scripts/stacks. v3.1 should move those provider-specific files into the plugin
directories and leave `scripts/provider-plugin` as a compatibility shim only
until every built-in provider is migrated.

## Provider Plugin Layout

Target built-in provider layout:

```text
plugins/providers/aws/
  egame-plugin.yaml
  commands/
    init
    plan
    up
    down
    start
    stop
    status
    ip
    ssh
    password
    login
  terraform/
    stacks/
    modules/

plugins/providers/local-qemu/
  egame-plugin.yaml
  commands/
    init
    plan
    up
    down
    start
    stop
    status
    ip
    ssh
  vagrant/
    Vagrantfile template/support
```

Core may provide helper commands/functions like:

- resolve instance JSON
- create/read per-instance workdir paths
- create/read per-instance Terraform data/state paths
- stream command output
- record provider/package operation state
- validate command JSON contracts

Provider plugins should call those helpers instead of duplicating path and state
logic.

## Provider State Model

Keep the semantic split:

- `up`: register/create provider resources. Disable once resources exist.
- `start`: power on an existing stopped VM. Disable when running.
- `stop`: power off an existing running VM. Disable when stopped/absent.
- `down`: deregister/destroy provider resources. This is the inverse of `up`.
- `delete`: remove the local instance registry entry. This is not a provider
  lifecycle command.

Rename TUI wording:

- Replace the current dangerous `Destroy` label with `Down` or a clearer pair:
  `Create` / `Destroy Provider Resource`.
- Avoid showing `Down` and `Delete` as peer buttons without explanation.
- Put `Down` in a Provider Debug/Danger area.
- Put `Delete local entry` below it with distinct copy explaining it only removes
  the local registry entry and does not destroy provider resources.

## TUI v3.1 Changes

### Provider Status Area

Add a provider-status area at the bottom of the upper-left panel:

- Run provider status once at TUI startup.
- Show each provider with configured/reachable state.
- Do not poll provider health constantly by default.
- Clicking a provider opens a provider menu.

Provider menu candidates:

- status/details
- doctor/provider check
- login/auth action if the provider plugin exposes one
- direct provider-specific admin action, such as TrueNAS host SSH, if exposed by
  the provider plugin

This should replace provider-specific Makefile concepts like `aws.login` and
`ssh.truenas` over time. Those actions should become provider plugin actions,
not hardcoded core/TUI behavior.

### Instance Lifecycle Area

Expose state-aware provider lifecycle controls:

- `Up`: visible/enabled only when provider resources are absent or unregistered.
- `Start`: visible/enabled only when provider resources exist and are stopped.
- `Stop`: visible/enabled only when provider resources are running.
- `Plan`: secondary action in lifecycle/debug area.
- `Init`: secondary action in lifecycle/debug area.
- `Provider Status`: secondary action, also useful after failed operations.
- `Reboot`: visible/enabled when provider is running, with confirmation.
- `Update`: visible/enabled when provider is running, with confirmation.
- `Show Password`: visible when the selected provider/OS exposes a password
  command or password capability. This is provider and OS dependent, not strictly
  Windows-only.

### Debug Panel

Add a collapsible or visually separated debug panel above provider down/delete:

- `Provider Status`
- `Plan`
- `Init`
- `Provision Wait`
- `Clear Provision State`
- `Restart Provision`
- `Recover Interrupted Operation`
- `Upload...`

Every destructive or state-resetting action needs confirmation.

`instance.recover` means: if local state says the last operation is still
`running` because the host process/TUI was interrupted, mark that operation
`failed`, set provider state to `error`, and record a recovery error. It does
not reach into the VM or provider; it only repairs local orchestration state so
the user can retry deliberately.

### Package Area

Add:

- `Reinstall` next to Install/Uninstall with destructive confirmation if needed.
- Clearer package lock/source labels:
  - bundle-selected packages are selected and locked by their bundle
  - directly selected packages can be removed
  - unsupported packages are disabled with reason

### Upload UX

Replace the coarse `upload` action with a TUI upload chooser:

- List direct subfolders under `upload/`.
- Let the user choose one or more subfolders.
- Upload selected subfolders to both Linux and Windows instances through
  provider/package-agnostic instance upload plumbing.
- The Makefile/script surface should accept selected upload names rather than
  blindly uploading every file under `upload/`.

## Makefile Cleanup

Remove:

- `all`

Reason: it hides important lifecycle steps (`init`, `up`, `ssh.wait`,
`provision`, `provision.wait`) behind one coarse target. The TUI should expose
and stream each step explicitly.

Eventually convert provider-specific host tasks into provider plugin actions:

- `aws.login`
- `ssh.truenas`

Keep CLI-only:

- `ssh.run`: future TUI command modal only if needed.
- test/lint/golden/dev maintenance targets.

## Provider Migration Order

1. Add provider plugin action metadata for provider-level actions.
2. Add core helper scripts for per-instance paths/state/env so plugins can call
   them without reaching into central provider internals.
3. Move `local-qemu` implementation into `plugins/providers/local-qemu/`.
   This is the smallest provider and a good proof of the new shape.
4. Move `gcp` next because it is newly added and smaller than AWS/TrueNAS.
5. Move `aws`, `vultr`, and `truenas` Terraform/Terramate stacks/modules into
   their provider plugin directories.
6. Move `raspberry-pi` metal implementation into its provider plugin directory.
7. Keep `scripts/provider-plugin` only as a compatibility shim or remove it once
   built-ins no longer need it.
8. Update tests to assert provider implementations no longer reference central
   provider-specific stack/module paths.

## Test Plan

- Unit/script tests:
  - provider manifests validate action metadata
  - provider dispatch can execute plugin-local commands
  - provider action list appears in TUI data sources
  - `instance.recover` remains local-state-only
  - upload selection rejects path traversal and missing folders
- Regression tests:
  - `make test`
  - provider dry-run matrix for every built-in provider
  - package action and reinstall dispatch
  - TUI `--check`
- Live smoke tests:
  - local-qemu Ubuntu baseline
  - one Terraform Ubuntu provider baseline, preferably GCP or TrueNAS
  - Vultr Windows baseline for password/show-password and remote GUI actions
  - optional `integration.packages` after baseline is stable

## Open Decisions

- TUI wording for provider resource deletion:
  - `Down` matches Makefile/provider contract.
  - `Destroy Provider Resource` is clearer but longer.
  - `Delete Local Entry` should replace plain `Delete` in dangerous UI sections.
- Whether provider actions should be a new manifest top-level key, e.g.
  `actions:`, or an extension under `commands:`.
- Whether `show-password` should become a standard provider command such as
  `password`, or a provider action only exposed when supported.
