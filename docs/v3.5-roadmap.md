# v3.5 Roadmap — Test suite performance (sub-15-minute CI)

Companion to [v3.4-roadmap.md](v3.4-roadmap.md).

v3.4 removed Ruby, shrank the slim runtime image ~5.6×, and cut the
dedicated Docker job to roughly 3.5 minutes. But it did **not** hit the
v3.4 acceptance bar of "a clean PR run completes required checks in less
than 15 minutes." The Ubuntu host job still runs about 23 minutes.

The cause is not the migration. Two large bash integration suites
dominate the host run, and Ubuntu GitHub runners are roughly 2× slower
than the macOS ARM runner on their subprocess-heavy workload:

| Suite | Lines | Ubuntu time | macOS time |
| --- | ---: | ---: | ---: |
| `scripts/test-plugins` | 1,205 | ~8m46s | ~4m28s |
| `scripts/test-instances` | 1,426 | ~8m02s | ~4m01s |

Both suites spawn hundreds of short-lived CLI processes (`./scripts/*`,
`jq`, `make`, inline Python) that each re-parse YAML, catalog, and plugin
data from cold. The wall-clock cost is cumulative process startup, not
any single expensive computation.

v3.5 makes the required PR run finish in under 15 minutes by amortizing
that startup cost and parallelizing the two suites — without losing any
of their current coverage.

## Goals

1. **Sub-15-minute clean PR run.** The slowest required leg
   (`test (ubuntu-24.04)`) completes in under 15 minutes excluding
   GitHub queue time.
2. **No coverage loss.** Every contract currently asserted by
   `test-plugins` and `test-instances` is still asserted after the
   migration. The two suites validate the plugin system contract and
   the instance model/state lifecycle respectively; both are
   load-bearing.
3. **One warm process per worker.** Catalog, plugin manifest, and
   defaults parsing happens once per test process, not once per
   assertion.
4. **Keep the rest of v3.4 intact.** Ruby stays gone, the slim image
   stays small, `make doctor`/TUI/`EVE_HOME` keep working.

## Current cost centers

| Area | Current cost | v3.5 direction |
| --- | --- | --- |
| `test-plugins` subprocess fan-out | ~77 `./scripts/*` + ~65 `jq` calls, nested dry-run dispatch loops, serial | Convert to pytest modules that import the Python SDK directly and reuse parsed catalog/plugin data within a process |
| `test-instances` subprocess fan-out | ~83 `./scripts/*` + ~52 `make` calls; `make` re-enters the Makefile and re-resolves env each time | Replace `make` round-trips with direct command/SDK calls in pytest; keep a thin smoke layer that still exercises the real Make targets |
| Serial suite execution | Suites run one after another in `scripts/test` | Run pytest with `pytest-xdist` (`-n auto`) so independent cases parallelize across cores |
| Cold catalog/plugin parsing | Every CLI invocation reloads YAML from disk | Parse once per worker via a session-scoped pytest fixture |
| Ubuntu runner slowness | ~2× macOS on subprocess workloads | Fewer, fatter processes reduce the multiplier's impact |

## Implementation order

### 1. Baseline and partition

- Record per-case timings inside `test-plugins` and `test-instances`
  so the slowest assertions are known before refactoring.
- Partition each suite's assertions into:
  - **pure** — can run against the Python SDK in-process (most cases)
  - **integration smoke** — must still shell out to a real `./scripts/*`
    or `make` target to prove the wiring
- Document the partition in `docs/v3.5-test-inventory.md`.

Acceptance:

- Inventory lists every current assertion with its target category.
- No assertion is dropped without an explicit, justified note.

### 2. Convert `test-plugins` to pytest

- Add `tests/python/integration/test_plugins.py` (and helpers) that
  cover the plugin-system contract by calling the Python SDK and CLI
  entry points in-process where possible.
- Use a session-scoped fixture to load catalog/plugin/manifest data
  once per worker.
- Keep a small `test-plugins-smoke` set of real subprocess calls for
  the wiring that genuinely needs a fresh process (shebang resolution,
  exit codes, stdout contracts other tools consume).
- Retire the bash `scripts/test-plugins` only after the pytest module
  asserts equal-or-greater coverage.

Acceptance:

- `make test.plugins` runs the pytest module (plus the smoke set).
- Coverage parity is documented against the v3.5 inventory.
- The suite's Ubuntu wall-clock drops materially.

### 3. Convert `test-instances` to pytest

- Same approach: `tests/python/integration/test_instances.py` covering
  instance create/delete/list/resolve/paths, `EVE_HOME` routing,
  registry/overlay/state paths, concurrent-create locking, bundle
  select/unselect, catalog override, and provider/OS support validation.
- Replace `make` round-trips with direct command calls except for a
  minimal smoke set that still proves the Make target routing works.
- Preserve the concurrent-create locking test — it caught real bugs in
  earlier milestones; port it using `multiprocessing`, not by deletion.

Acceptance:

- `make test.instances` runs the pytest module (plus the smoke set).
- Concurrency and golden-env assertions are preserved.
- The suite's Ubuntu wall-clock drops materially.

### 4. Parallelize the Python test run

- Add `pytest-xdist` and run `pytest -n auto` in `scripts/test-python`
  (and any suite-specific runners).
- Make fixtures worker-safe: per-worker temp registries/state dirs, no
  shared mutable global files.
- Keep cache misses and skips explicit — no silent test skipping under
  parallelism.

Acceptance:

- `make test` on Ubuntu finishes under 15 minutes.
- Tests are deterministic under `-n auto` (no order-dependent flakes
  across at least 3 consecutive runs).

### 5. Re-verify CI contract

- Update `scripts/test-ci-pipeline` if suite registration changes.
- Update `docs/v3.4-acceptance.md`'s timing row (or a new
  `docs/v3.5-acceptance.md`) to PASS with the new numbers.
- Confirm the required-check set is unchanged:
  `test (macos-latest)`, `test (ubuntu-24.04)`, `docker-runtime-slim`.

Acceptance:

- A clean PR run shows the Ubuntu job under 15 minutes in CI.
- No required check was removed to hit the target.

## Non-goals

- Reducing test coverage to hit the time budget. Speed comes from
  amortizing process startup and parallelism, not from deleting checks.
- Removing the integration smoke layer entirely. Some assertions must
  still exercise a real subprocess to prove wiring; keep a thin set.
- Rewriting guest-side provisioning tests.
- Changing provider behavior, Terraform/Terramate/Vagrant/QEMU, or
  cloud credentials.
- Splitting CI into more required jobs as the primary lever; in-process
  amortization comes first, parallelism second, job-splitting only if
  still needed.

## Acceptance

- A clean PR run completes required checks in under 15 minutes,
  excluding GitHub queue time, with `test (ubuntu-24.04)` as the
  measured slowest leg.
- `test-plugins` and `test-instances` coverage is preserved, documented
  against `docs/v3.5-test-inventory.md`.
- The Python test run is parallel (`pytest -n auto`) and deterministic
  across at least 3 consecutive runs.
- Required PR checks are unchanged.
- `make test` is green on macOS and Ubuntu.
