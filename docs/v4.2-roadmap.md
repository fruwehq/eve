# v4.2 Roadmap — Plugin source management, lean test surface, and package-repo hygiene

Status: proposed (2026-06-17). Follows v4.1 (agnostic host shipped + merged to `main`;
all plugins external; sibling repos public with their own conformance CI).

## Philosophy — the host stays empty; the user composes it

v4.0/v4.1 made eve a provider/package-agnostic host: core ships **no** providers,
packages, or catalog. v4.2 makes that emptiness *usable*. Today a fresh clone has no
plugins and `config/plugin-sources.yaml` points at local sibling checkouts — fine for
development, opaque for a real user. The fix is **not** to ship a default plugin (that
re-introduces the coupling we deliberately removed); it is to give the user first-class
**plugin-source management** — CLI, config, and TUI — plus a **recommended list** they
can opt into in one step.

The other v4.2 threads are hygiene: shrink a test matrix that no longer earns its cost
now that core is Python, finish the os-portability decoupling, and tidy where packages
live (AI/dev packages move out of the generic Linux repo).

## Goals

1. A new user can discover, add, and remove plugin sources without hand-editing YAML —
   from the CLI **and** the TUI — with a curated "recommended" set offered, never
   auto-installed.
2. Zero default plugins; **no circular dependency** between eve and any plugin repo.
3. A leaner, cheaper test matrix: Ubuntu is the full run; macOS is a small
   host-portability job; eve-as-container CI/image is removed (per v4.1 §3).
4. `test-os-portability` coverage lives where its subject lives.
5. Packages live in the right repo: AI/dev packages move to `eve-plugins-ai`.
6. Documentation a newcomer can actually follow.

---

## 1. Plugin source management (CLI + config + TUI + recommended list)

**No default plugins.** `config/plugin-sources.yaml` ships empty (or with everything
commented). Core never depends on a plugin repo — the recommended list is *data*, not a
dependency.

**Config.** `plugin-sources.yaml` stays the source of truth (id, url, ref, auth).
`.eve/plugin-sources.yaml` overlays it (user sources win by id), as today.

**CLI.** First-class management verbs (wrapping the existing `eve pull`/`plugins-sync`):
- `eve plugin source list` — configured sources + sync status
- `eve plugin source add <url> [--ref <tag>] [--id <id>]`
- `eve plugin source remove <id>`
- `eve plugin source recommended` — print the curated catalog (below)
- `eve plugin source add --recommended <id>` — add a known-good source by short id
- `eve pull` — unchanged (materialize configured sources)

**Recommended list (the "registry", folded in).** A small, curated, **first-party**
catalog of known plugin repos shipped as static data in core (e.g.
`config/recommended-sources.yaml`): `eve-providers`, `eve-packages-linux`,
`eve-packages-windows`, `eve-plugins-ai`, future `eve-plugins-openwrt`. Each entry: id,
description, public git URL, suggested ref, tags. This is the lightweight external
registry from the v4.0 roadmap — a recommendation surface, **not** an install default
and **not** a network service. It can later point at a hosted index, but static data
ships first.

**TUI.** A plugin-management screen:
- list configured sources with synced/stale state
- add a source by URL, or pick from the recommended list (with descriptions)
- remove a source
- trigger `pull` and show progress
- surface recommended options prominently on an empty install ("You have no plugins —
  add the recommended providers/packages?")

**Pinning/auth.** Recommended entries use public URLs with `auth: none`; keep ref
pinning (tags) as the encouraged default once the sibling repos cut releases.

**Acceptance:** fresh clone → `eve` (TUI) → add recommended → `pull` → catalog
populated, with no YAML editing and no plugin bundled in core.

## 2. Lean test surface

Now that core is Python (cross-platform by default) the old "run everything on every
OS" matrix is mostly redundant. Reduce it to what actually catches OS-specific bugs.

**2a. Ubuntu = the full run.** Keep the complete `make test` on `ubuntu-latest`.

**2b. macOS = a slim host-portability job, not the full suite.** eve is a *host*
orchestrator and real users run it on macOS (BSD vs GNU coreutils, `dscacheutil` vs
`getent`, `sed`/`date`/`realpath`/`mktemp` differences). That surface is real but small.
Replace the full macOS `make test` with a focused job running only the host-portability
slice: `test-os-portability`, `test-doctor`, `dispatch`/`check-required`/`remote-*`
launcher syntax — the scripts that actually branch on `Darwin`. Drop the macOS re-run of
OS-agnostic Python logic already covered on Ubuntu.

**2c. Remove eve-as-container CI + image (confirm v4.1 §3).** v4.1 decided eve is **not**
distributed as a container and "eve in docker" is unsupported (it breaks
local-qemu/vagrant/GUI launchers). Yet `docker-runtime-slim` + `docker-runtime-full` CI
jobs and `docker/Dockerfile*` still build `eve/eve:slim|full`. **Decision to confirm:**
if eve-in-container stays unsupported, remove those CI jobs, `docker/` runtime image
build, and the `docker.runtime.*` make targets. (The *docker provider* — containers as
instances, v4.1 WS5 — is unrelated and stays.) This is the single biggest CI-time win.

**Net:** one full Ubuntu run + one small macOS portability job; no Docker image build.

## 3. `test-os-portability` — coverage where its subject lives

`test-os-portability` currently only exercises a *sibling's* shell helper
(`eve-providers/.../raspberry-pi/lib/host-resolver.sh`); decoupled, it skips and tests
nothing. Two parts:
- **Sibling-specific logic** (the rpi host-resolver getent/dscacheutil dispatch) moves
  into **eve-providers' own** test suite/CI, where that code lives.
- **Core host-portability** (any OS-branching in core scripts) gets a hermetic test in
  eve using fixtures, folded into the 2b macOS job. Remove the dead skip from eve.

## 4. Package-repo structure — `commands/` vs `provision/` (clarify, keep)

Observed asymmetry in the package repos (e.g. `eve-packages-windows/discord/`):
- `provision/windows/discord.ps1` — **guest-side** install steps (PowerShell, run *on the
  box* by the provisioning pipeline; reboot/state aware).
- `commands/windows/{status,down}` — **host-side** dispatch entrypoints (`#!/usr/bin/env
  sh`, run *on the user's host* by eve; they remote-exec a quick query/removal).

This is two genuinely different execution contexts, which is why the file types differ
(host sh vs guest ps1) and why `install` flows through `provision/` (heavy, staged) while
`status`/`down` are lightweight `commands/`. The split is sound — folding status/down
into the provisioning pipeline would over-engineer one-shot operations.

**v4.2 action: document the convention, don't restructure.** Add a short "anatomy of a
package" doc (host commands vs guest provision; sh vs ps1; when to use which) to the
package repos and the plugin contract. Optionally revisit naming only if the doc proves
insufficient. No code move planned.

---

## 5. AI/dev packages → `eve-plugins-ai`

`eve-packages-linux` currently mixes generic Linux packages with AI/dev tooling. Move the
AI/dev set to the existing `eve-plugins-ai` repo:
- **AI agents:** `claude`, `codex-cli`, `goose`, `hermes`, `opencode`
- **Dev tooling:** `dev-toolchain`, `vscode`

Stays in `eve-packages-linux`: desktops (gnome/kde/xfce/*-headless), remote access
(rdp/vnc/rustdesk/nomachine/splashtop/thinlinc/waypipe/xpra/sunshine), `steam`,
`discord`, `docker`, `macos-desktop-theme`.

Make `eve-plugins-ai` **public** once populated (like the other siblings, `auth: none`),
give it the same conformance CI, and add it to the recommended list (§1). Open question:
keep `vscode`/`dev-toolchain` under `eve-plugins-ai` or split a future
`eve-plugins-dev`? Default: `eve-plugins-ai` now, split later only if it grows.

---

## 6. Documentation

- Per-repo READMEs: what the repo is, how to consume it, how to contribute a plugin.
- A plugin-authoring guide keyed off `eve-plugin-template` (the §4 "anatomy" doc).
- User quickstart: clone → add recommended sources (§1) → pull → create instance.
- Keep eve's README (refreshed at v4.x) as the entry point; link out to the rest.

---

## Non-goals

- Shipping any plugin bundled in core (would re-introduce the coupling v4.0/v4.1 removed).
- A hosted/network plugin registry service (the recommended list is static data; a hosted
  index can come later behind the same UX).
- Restructuring the `commands/`/`provision/` split (documenting it is enough).
- Supporting eve-in-a-container (explicitly out, per v4.1 §3).

## Sequencing (indicative)

1. §2c remove eve-as-container CI/image (fast, independent, biggest CI win).
2. §1 plugin-source CLI + config + recommended list, then TUI.
3. §5 AI/dev migration to `eve-plugins-ai` (+ public + conformance CI + recommended entry).
4. §3 os-portability relocation; §2a/2b matrix trim.
5. §4/§6 documentation.
