## TASK 1 ‚Äî Local-folder plugin sources in the TUI (roadmap Bug B)

**Why:** this is what lets the human (and you) test with plain `eve tui` instead
of env vars. Highest priority.

**Backend is already done** ‚Äî do not reinvent it:
- `eve_sdk/registry.py::resolve_url` treats a non-URL string as a local path
  (absolute, or relative to the repo root) ‚Äî a first-class source kind.
- `tui/plugins.py::add_url` already accepts such a path as `url`.
- `registry.add_source` adds/replaces by id; `remove_source` removes.

**Build the TUI surface** (screen lives in `tui/`, helpers in `tui/plugins.py`;
find the plugin-source screen via `grep -rn "configured_rows\|recommended_rows" tui/`):
1. List configured sources (`configured_rows()`), clearly distinguishing
   **local-folder** entries (url is a filesystem path) from remote ones. The
   list should be usable to drive everything from local folders instead of the
   recommended remote repos.
2. An **Edit** screen for a single source entry: fields id, url/folder, ref,
   subdir, auth. Saving calls `add_source` (replace-by-id). Validate the folder
   exists when it's a local path.
3. An **Add local folder** flow (path input ‚Üí `add_url`).
4. **Default action on a custom (non-recommended) entry opens its Edit screen.**
   Recommended-catalog entries keep their add/remove behavior.
5. After mutations, the existing Pull/prune flow materializes them.

**Tests:** extend `tests/python/test_tui_*` and/or `tui/plugins.py` helper tests
to cover: rendering a local-folder source, the edit round-trip (load ‚Üí change ‚Üí
save ‚Üí `add_source` called with the right dict), default-action routing to edit
for custom vs. add for recommended. Keep them hermetic.

**Acceptance:** from a clean config, a user can, entirely in `eve tui`, add a
local folder as a plugin source, edit it, pull it, and see its providers/packages
appear ‚Äî then `eve tui` operates with no env vars. Update roadmap Bug B to ‚òë.

## TASK 2 ‚Äî QEMU provisioning, end-to-end (roadmap Bug D)

Free and local ‚Äî do this before Vultr. Goal: a full lifecycle works from the CLI
and the TUI.

1. Configure local sources per the roadmap "How to test" (providers `../eve-providers`

**Acceptance:** `create ‚Üí up ‚Üí provision ‚Üí status ‚Üí down` works for QEMU from
