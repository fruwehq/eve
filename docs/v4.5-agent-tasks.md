## TASK 1 — Local-folder plugin sources in the TUI (roadmap Bug B)

**Why:** this is what lets the human (and you) test with plain `eve tui` instead
of env vars. Highest priority.

**Backend is already done** — do not reinvent it:
- `eve_sdk/registry.py::resolve_url` treats a non-URL string as a local path
  (absolute, or relative to the repo root) — a first-class source kind.
- `tui/plugins.py::add_url` already accepts such a path as `url`.
- `registry.add_source` adds/replaces by id; `remove_source` removes.

**Build the TUI surface** (screen lives in `tui/`, helpers in `tui/plugins.py`;
find the plugin-source screen via `grep -rn "configured_rows\|recommended_rows" tui/`):
1. List configured sources (`configured_rows()`), clearly distinguishing
   **local-folder** entries (url is a filesystem path) from remote ones. The
   list should be usable to drive everything from local folders instead of the
   recommended remote repos.
2. An **Edit** screen for a single source entry: fields id, url/folder, ref,
   subdir, auth. Saving calls `add_source` (replace-by-id). Validate the folder
   exists when it's a local path.
3. An **Add local folder** flow (path input → `add_url`).
4. **Default action on a custom (non-recommended) entry opens its Edit screen.**
   Recommended-catalog entries keep their add/remove behavior.
5. After mutations, the existing Pull/prune flow materializes them.

**Tests:** extend `tests/python/test_tui_*` and/or `tui/plugins.py` helper tests
to cover: rendering a local-folder source, the edit round-trip (load → change →
save → `add_source` called with the right dict), default-action routing to edit
for custom vs. add for recommended. Keep them hermetic.

**Acceptance:** from a clean config, a user can, entirely in `eve tui`, add a
local folder as a plugin source, edit it, pull it, and see its providers/packages
appear — then `eve tui` operates with no env vars. Update roadmap Bug B to ☑.

## TASK 5 — Bundles as plugins (K) → gaming-streaming (I) → wizard compat (G)

Do these as one coherent arc; **K is the foundation**.

**5a — `kind: bundle` + dependencies (roadmap K).**
- Add `bundle` to the plugin-kind allow-list (`eve_sdk/plugin_manifest.py` ~L254:
  `{"provider","package","os"}` → add `"bundle"`) and to
  `core/schema/plugin-manifest.schema.json` (kind enum + a `bundle` manifest
  branch). Keep core agnostic — `test-core-boundary` green.
- Define the `kind: bundle` manifest: an id + the member packages it groups,
  modeled as a general **`requires:`** (dependency) list. Add `requires:` as a
  capability packages may also use.
- **Remove `bundles:` from package manifests + schema.** Migrate every existing
  bundle (gaming-streaming, desktop-gnome, desktop-gnome-mac, kde/xfce + headless
  variants, …) into `kind: bundle` plugins in their owning repo.
- Catalog aggregator + `catalog_view.py` build `bundles` from `kind: bundle`
  plugins; compute each bundle's OS-compat from its members
  (`installable_os_families` or similar on each bundle row).
- Keep dependency resolution minimal (expand bundle → member package ids; honor
  package `requires`); note anything deeper as future work.

**5b — gaming-streaming Windows bundle (roadmap I).** Define `gaming-streaming`
as a `kind: bundle` plugin in eve-packages-windows, members `[steam, sunshine,
rustdesk]`; drop the Ubuntu `steam` membership. It must resolve windows-only.

**5c — wizard differentiation (roadmap G).** In the new-instance wizard
(`tui/widgets.py`), **visually differentiate** incompatible packages *and* bundles
for the selected OS (dim/distinct color); if clean differentiation isn't feasible,
add a **show/hide incompatible** toggle, **hidden by default**. Don't let
incompatible items be selected. Verify `gaming-streaming` shows compatible for
`windows-server-2025` and incompatible/hidden for Ubuntu.

**Tests:** kind:bundle schema + load, bundle OS-compat in `catalog_view`, package
`bundles:` removal, wizard differentiation. This spans eve (schema/sdk/tui) and
the package repos (bundle plugins) — commit to `v4.5` in each.
