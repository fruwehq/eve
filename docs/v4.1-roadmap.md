# v4.1 Roadmap — Easy adoption, host-tool awareness, the docker provider, and on-instance content state

Companion to [v4.0-roadmap.md](v4.0-roadmap.md). Where v4.0 makes the core
**provider/package-agnostic** (external registry, resolver, conformance harness,
agnostic core), v4.1 is about **adoption and reach**: make eve trivial to install
on macOS and Ubuntu, make its host-tool dependencies self-describing, add a
**docker provider** (containers as instances), and let packages run **from inside
an instance** via a thin on-box runner with **content state living on the
instance** — which is what makes an edit-config → rebuild loop first-class.

## Philosophy — eve is a thin orchestrator over existing tools

The through-line for every decision below: **eve orchestrates existing tools; it
is not a configuration-management system.** This is already true for infra —
Terraform/Terramate own infra stand-up and eve just drives them. v4.1 extends the
same stance to the package tier:

- Packages stay **readable shell/PowerShell** driven by a **minimal runner**.
- **No Ansible, no convergence engine, no drift detection.** A package is a small
  ordered set of steps a human can read; the runner executes them.
- The "brain" (catalog, resolver, dependency graph, config) stays **host-side**;
  the guest only ever runs flattened, explicit steps.

Keeping the surface thin is a feature: it's what keeps packages authorable by
anyone and keeps eve debuggable.

## Goals

1. **Zero-setup install** for the two target audiences (macOS + Ubuntu users, and
   plugin authors).
2. **Self-describing host-tool dependencies** — eve knows which host tools it needs
   from the providers you pulled, and `eve doctor` tells you what's missing.
3. **A docker provider** — containers as cheap, throwaway instances (also the
   natural CI substrate for `eve-packages-*`).
4. **`eve-local-install`** — run a package install *from inside* an instance via a
   thin per-OS runner, with **desired + observed content state on the instance**.
5. **Edit-config → rebuild loop** as a first-class operation, enabled by (4).

## 1. Distribution / zero-setup

Primary install paths, chosen for the lowest-friction adoption on each platform:

- **Ubuntu:** `pipx install eve` (isolated, no venv juggling, standard on modern
  Ubuntu).
- **macOS:** a **Homebrew tap** (`brew install fruwehq/tap/eve`).

**No `uv`** as a primary path — it's uncommon on macOS and would add a setup step
for the exact audience we want to reach. (Contributors keep using poetry for dev;
this is about *end-user/author* install.)

The bar: a macOS or Ubuntu user, or a plugin author, can go from nothing to a
working `eve` with **one command** and no Python-environment knowledge.

## 2. Host-tool dependencies — never bundle, always derive

Core ships **no** provider tools. The required host toolchain is **derived from the
providers the user actually pulled**:

- a `terraform`-engine provider needs `terraform` + `terramate`,
- a `qemu`-engine provider needs `qemu`,
- a `vagrant`-engine provider needs `vagrant`,
- and a plugin may declare **`requires.tools`** in its manifest for extras (e.g. a
  cloud CLI like `aws`/`gcloud`/`vultr-cli`).

**`eve doctor` becomes toolchain-aware:** it reports only the tools the pulled
providers/packages actually require, with install hints, and is green when those
are present — instead of a static list of every tool eve ever might use. This
builds directly on v4.0's `requires` block and the engine metadata already in
provider manifests.

## 3. Docker — strictly the docker *provider*, not an eve runtime

Docker appears in eve **only** as a provider engine: it stands up a **container as
a (limited Linux) instance**, a peer of qemu/vagrant — cheap, throwaway, and the
natural substrate for `eve-packages-*` CI.

eve is **not** distributed or run *inside* a container for normal use. Running eve
in a container breaks local-qemu, vagrant, and the `remote-*` GUI launchers (which
need the host's display/network), so there is no "eve in docker" runtime. (The
v4.0/Phase-2 `eve-docker` toolchain wrapper was retired for exactly this reason;
the CI runtime images and the contributor devcontainer are unrelated to how end
users run eve.)

The docker provider is **independent of and unrelated to** `eve-local-install`
(§4) — they solve different problems and must not be coupled.

## 4. `eve-local-install` — thin on-instance runner, host-side brain

Let a package install run **on the instance side, executed on that side** (mirroring
what eve does today from the host, just locally on the target). A Dockerfile
`RUN eve install <pkg>` at image-build time is one application; it is not the only
one.

The split that keeps this honest:

- **On the instance:** a **thin, self-contained per-OS runner** — generalize the
  existing `runner.ps1` (Windows) to Linux. It needs only the box's **native
  shell**; it is **not** a full eve/Python install on the instance.
- **On the host:** all orchestration (catalog, resolver, dependency graph) stays
  put. The host ships the instance a **flattened, readable step bundle + config**
  that the runner executes.

This preserves the thin-orchestrator philosophy: the box runs explicit steps; the
host owns the logic. (Terminology: v4.1 may reword v4.0's "SSH-readiness" boundary
to "**manageable**", since a target can be reachable without literal SSH — e.g.
`docker exec`.)

## 5. State — single owner per tier

State is partitioned by tier with **exactly one owner each**, no shared mutable
state:

- **Content state (desired + observed)** for a package on an instance lives **on
  the instance** (written/read by the on-box runner).
- **Infra / lifecycle state** stays **host-side** (as today — Terraform state, the
  instance registry).

Putting **desired** content-state on the instance is what makes the
**edit-config → rebuild** loop first-class: edit the instance's desired config,
re-run the runner, and it reconciles to the new desired steps — without the host
having to carry per-instance content state.

## 6. `.eve/` folder convention

(Carried from the original v4.1 notes.) A project carries a `.eve/` folder holding
everything needed to run it across platforms. Discovery is **not** `.git`-style
walk-up: eve uses the **selected directory** or that directory's **`.eve/`
subdir**, with an explicit **marker** designating an eve folder. Consumer projects
tuck eve content into `.eve/`; provider/package repos (where the plugins are the
attraction) expose them as **visible subdirs**, not a dotfolder — so v4.0's
per-subdir source pinning is unaffected.

## Non-goals

- A configuration-management system. No Ansible, no convergence/drift engine; the
  package tier stays readable shell/PowerShell with a minimal runner.
- Running eve itself as a container image / "eve in docker" runtime (§3).
- Bundling provider tools into core (§2) — they are derived, never vendored.
- Full eve/Python on managed instances (§4) — the on-box footprint is the thin
  runner only.

## Sequencing (indicative)

| Order | Work | Why here |
| --- | --- | --- |
| 1 | Distribution: pipx + Homebrew tap | Unblocks real adoption; independent of the rest |
| 2 | Host-tool awareness in `eve doctor` (derive from pulled providers + `requires.tools`) | Builds on v4.0 `requires`; small, high-value |
| 3 | Docker provider (containers as instances) | Cheap CI substrate; standalone engine |
| 4 | `eve-local-install` runner (generalize runner.ps1 to Linux) + flattened step bundle | The on-box execution model |
| 5 | On-instance content state + edit-config → rebuild loop | Depends on (4) |
| 6 | `.eve/` folder convention + "manageable" terminology | Ergonomics; touches discovery |
