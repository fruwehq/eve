# v3.1 Roadmap

Companion to [v3.1-provider-plugin-boundaries-plan.md](v3.1-provider-plugin-boundaries-plan.md).
That document describes the *target* state. This one tracks *what is left to do*
and *in what order*, based on the current state of the tree.

Two concerns drive everything below:

1. **Fragility** â€” regressions are caught on live VMs, not in CI.
2. **Plugin-driven, not plugin-shaped** â€” provider/package boundaries exist in
   manifests but central scripts still hardcode `aws`, `gcp`, `truenas`,
   `vultr`, `local-qemu`, `raspberry-pi`.

The roadmap is split into two layers: tactical work that finishes the v3.1 plan,
and architecture work that must precede most of it.

## Part 1 â€” Architecture (do this first)

The original plan is a *layout* plan. It does not establish the core/plugin
contract, the language story, the schemas, or the state-ownership rules that
the rest of v3.1 depends on. Without these, the plugin relocation just moves
hardcoded coupling from one place to another.

### A. Core/plugin SDK

All six `plugins/providers/*/commands/provider-command` files are
byte-identical 95-line Ruby wrappers that `exec` back into root-level
`scripts/tf-*`, `scripts/start`, `scripts/instance-ssh`. There is no defined
"public API a plugin is allowed to call."

- Introduce `core/sdk/` (or `lib/eve/`) with a versioned surface:
  - `resolve` â€” parse `EVE_RESOLVED_JSON` into a typed struct
  - `workdir` â€” per-instance paths (state, tf, logs, uploads)
  - `state` â€” read/write observed state, operation history (sole owner)
  - `log` â€” structured streaming output
  - `ssh` â€” provider-agnostic SSH (after the access model lands)
  - `contract` â€” validate command in/out JSON
- Replace each plugin's 95-line wrapper with `require 'eve/sdk'; Eve::Provider.dispatch(ARGV)`.
- Document the SDK boundary explicitly. Nothing else in `scripts/` is part of
  the contract.

### B. Schemas for the central data contracts

`EVE_RESOLVED_JSON` is consumed by every plugin command, the TUI, and
observed-state writers. There is no JSON schema. Today `provider-command` does
`resolved.fetch("machine").fetch("provider")` and explodes at runtime when a
field moves.

- Author JSON schemas:
  - `core/schema/resolved-instance.schema.json`
  - `core/schema/observed-state.schema.json`
  - `core/schema/plugin-manifest.schema.json`
  - `core/schema/command-io.schema.json` (the in/out shape of plugin commands)
- Validate at every boundary: resolver writes â†’ dispatcher validates â†’
  plugin consumes.
- Add `make schema.check` as a first-class test.
- Schema-version the manifest. Validate `api_version` on load. Refuse unknown
  versions instead of silently ignoring them.

### C. State ownership

State today is scattered: `.eve/instances.yaml`, per-instance JSON,
observed-state cache (same JSON, nested), per-provider tfstate, operation
history, TUI in-memory snapshots. Each consumer opens, mutates, and writes
the JSON differently. The TUI refresh races with CLI status reads on the same
file.

- Make `core/sdk/state` the only thing allowed to read/write
  `.eve/state/instances/<name>.json`. Every other caller goes through it.
- Add file locking. The race exists today.
- Surface a single "view object" the CLI and TUI both render from, so
  `make status` and the TUI cannot disagree about what "running" means.

### D. Language consolidation

Three languages with no rule for which is used where:

- bash: `start`, `stop`, `status`, `tf-*`, `provision`, all `test-*`, all
  `truenas-*`
- ruby: `instance-resolve` (322 lines), `provider-dispatch`,
  `package-dispatch`, `provider-command`
- python: `eve-tui` (3,226 lines)

The same resolved JSON is parsed three different ways with three different
bug surfaces.

- Core language: Ruby (it already holds the serious orchestration).
- Bash: allowed only inside `plugins/*/provision/` (guest-side shell).
- Python: allowed only for the TUI (Textual is Python).
- Stop accepting new bash in `scripts/`.

### E. Capability vs action vs supports

The manifest mixes three concerns and the TUI compensates with hardcoded
checks (`provider == "vultr"`, `provider == "raspberry-pi"`).

- **capabilities**: behavioral flags (`password`, `needs_provider_ip`,
  `host-ssh`)
- **actions**: UI buttons the TUI renders from manifest
- **supports**: OS families and engines

Add capabilities to provider manifests; replace the hardcoded TUI branches
with manifest reads.

### F. TUI as view layer only

`eve-tui` has functions like `effective_provider_state`,
`should_apply_live_provider_state`, `aggregate_state`,
`normalize_provider_state`. These are **state-machine rules**, not UI
rendering. They belong in the core SDK and must be shared with the CLI.

- Extract the state-machine logic into core (Ruby).
- The TUI calls it via JSON-stdio (e.g. `scripts/instance model --instance foo`
  returns the same view object the TUI renders).
- The TUI becomes a pure view layer.

### G. Provisioning protocol

The Ubuntu/Windows runners under `oses/<id>/provision/scripts/` are real
state machines but the host and guest have no shared contract. The host
uploads steps; the runner has no idea what the host expected; failures
require `provision.clear-state` and re-upload. Live logs are the only feedback
channel.

- Define a "provision protocol":
  - host writes a manifest of steps with expected hashes
  - runner emits structured status (`{step, phase, started_at, ...}`) into a
    JSON file the host scrapes
- Then `provision.wait` shows live progress, not tailed logs, and CI can
  dry-run the manifest without a VM.

### H. Core boundary lint

Nothing prevents a new contributor from adding `case PROVIDER in foo)` to a
central script. That is how the current leaks accumulated.

- Add `scripts/test-core-boundary`. Greps `scripts/` for provider/OS IDs and
  fails on hits outside an allowlist of dispatchers.
- Same lint for OS IDs.
- The original plan's "provider implementations no longer reference central
  provider-specific stack/module paths" becomes a real, enforced test.

## Part 2 â€” Tactical (finish what the v3.1 plan promised)

### 1. Plugin boundaries are still a manifest illusion

| Script | Hardcoded provider refs |
|---|---|
| `scripts/start` / `scripts/stop` | 16 each (aws / gcp / vultr / truenas / raspberry-pi cases) |
| `scripts/status` | 16 |
| `scripts/instance-resolve` | 13 (OS-key validation per provider) |
| `scripts/instance-ip` | 12 |
| `scripts/tf-env` | 11 (provider-specific `TF_VAR_*` emission) |
| `scripts/profile-resolve` | 10 |

Plus four TrueNAS-only helpers (`truenas-cloudinit-upload`,
`truenas-cloudinit-delete`, `truenas-dataset-verify`,
`truenas-vm-disk-prepare`) sit in central `scripts/`.

Tasks:

1. Move `start`/`stop`/`status`/`ip`/`ssh` implementations *into* each
   provider plugin. Delete the `case` blocks. Central scripts become 5-line
   dispatchers.
2. Move `tf-init`/`tf-plan`/`tf-apply`/`tf-destroy` into a shared
   `plugins/providers/_terraform-common/` helper (or inline into the four
   cloud plugins). Same for `vagrant-up`/`vagrant-destroy` into `local-qemu`.
3. Move the four `truenas-*` scripts into `plugins/providers/truenas/`.
4. Move provider-specific OS-key validation out of `instance-resolve` into
   per-plugin `validate` hooks.
5. Move provider-specific `TF_VAR_*` mapping out of `tf-env` into each
   provider's plugin.
6. Land the local-qemu and raspberry-pi relocations the plan marks
   "intentionally deferred".
7. Delete the legacy `scripts/provider-plugin` shim once nothing references
   it.

Acceptance criterion: removing `plugins/providers/aws/` disables AWS
entirely, with zero edits to central scripts. That is the test for "really
plugin-driven."

### 2. Tests catch almost nothing pre-VM

What exists: shell suites validating manifests, golden-env snapshots,
terraform `validate`, shellcheck, a TUI `--check`. Recent commits
(`Fix Ubuntu provisioning log ownership`, `Force Windows RustDesk password
marker refresh`, `Fix Ubuntu RustDesk LightDM preseed noise`,
`Fix Ubuntu provision step ordering on empty arrays`) all show regressions
caught only on live VMs.

Gaps:

- **No pytest.** `scripts/eve-tui` is 3,226 lines / 34 functions / one
  file with no unit tests beyond a `runpy` smoke check.
- **No fake provider plugin** that exercises the full
  `up â†’ ssh.wait â†’ provision â†’ status â†’ stop â†’ down` loop in CI against a
  tempdir. `tests/fixtures/plugins/providers/acme/` exists only as a
  manifest fixture.
- **No provision-runner tests.** The Ubuntu and Windows runners have zero
  direct tests; failures appear as live-VM regressions.
- **`scripts/test-instances` (1045 lines) and `scripts/test-plugins`
  (928 lines)** are huge monolithic bash scripts; failures are hard to
  localize.

Tasks:

1. Add pytest under `tests/python/`. Extract pure helpers from `eve-tui`
   into an importable module and unit-test them.
2. Build `tests/fixtures/plugins/providers/fake/` that completes the full
   lifecycle against a tempdir. Write `scripts/test-lifecycle` exercising
   up/start/stop/down + observed-state cache against it.
3. Containerized provision-runner test: feed canned step lists into
   `oses/ubuntu-26.04/provision/scripts/runner.sh` inside an Ubuntu
   container, assert state transitions and log output. Same idea for the
   PowerShell runner where practical.
4. Split `test-instances` and `test-plugins` into per-concern suites.
5. Add a `provision-runner` smoke job to CI so the runner state machine is
   exercised on every PR.

### 3. TUI

`scripts/eve-tui` â€” 3,226 LOC, single file. Still contains five hardcoded
provider-name checks (`provider == "vultr"` password gate,
`provider == "raspberry-pi"` provider-IP flow).

Tasks:

1. Split into a package: `tui/state.py`, `tui/widgets/`, `tui/commands.py`,
   `tui/render.py`, etc.
2. Replace hardcoded provider checks with manifest capability flags from
   architecture Â§E.
3. Add pytest coverage for state-machine functions *before* splitting (to
   lock in behavior).
4. After architecture Â§F lands: convert the TUI to a pure view over the
   shared core view object.

### 4. Plugin action metadata

7 of 24 package plugins declare `actions:`. Zero providers do, despite the
plan calling for `init`/`plan`/`status details`/`login` to be
provider-declared actions.

Tasks:

1. Add `actions:` to provider manifests (`init`, `plan`, `status-details`,
   plus `login` for AWS/GCP, `host-ssh` for TrueNAS).
2. Audit the 17 packages without `actions:`. Desktops/toolchains genuinely
   have none; `rustdesk`, `xpra`, `thinlinc`, `rdp`, `waypipe`, `discord`,
   `steam`, `vscode` likely do.
3. Render provider actions in the TUI debug panel from manifest, removing
   the remaining hardcoded `aws.login` / `ssh.truenas` Makefile targets.

### 5. VM identity model

The plan describes a `bootstrap_user` / `provision_user` / `human_user`
triple. Today the resolved JSON still emits a flat `VM_USER_NAME` with
provider-specific fallbacks (`profile-resolve` line 258 hardcodes the
TrueNAS provision user).

Tasks:

1. Add `access: {bootstrap_user, provision_user, human_user}` to resolved
   instance JSON; each provider plugin populates it from its own rules.
2. Migrate Linux provision steps that use `$USER` to write to `human_user`
   explicitly while running privileged commands as `provision_user`.
   (Plan's "migration caveat" â€” gnarliest piece because RustDesk/VNC/desktop
   autologin all touch `$USER`.)
3. Drop the `ubuntu` / `Administrator` image-default fallbacks once
   existing instances are recreated.

### 6. Smaller cleanups (already in the plan)

- Finish the `PROFILE â†’ INSTANCE_NAME` rename in internal variables and log
  messages.
- Confirm `.eve/config.yaml` precedence is enforced in `config-env`.
  Structured config defaults landed but TUI editing is not exposed.
- TUI wording for "Down" vs "Delete local entry".
