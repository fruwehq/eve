# v4.5 roadmap — operationalize the agnostic core

## Goal

Milestone 4 (v4.0–v4.4: the plugin-agnostic-core effort) was merged but **never
validated at runtime**. The §9/§14 refactors (terramate stacks + OS assets moved
out of core) broke the terraform runtime, and there is no end-to-end proof that a
plain user can provision and operate an instance.

**v4.5's job is to make milestone 4 actually work** — proven by minimalistic real
usage: provisioning a Vultr instance and a local QEMU instance, and driving the
full lifecycle from the TUI (`eve tui`). Every bug found during that usage is
fixed here and recorded below.

## Hard constraints

- **No `EVE_PLUGIN_ROOTS` / `EVE_PLUGIN_ROOTS_EXCLUSIVE` for testing.** Env-var
  overrides make the human and the agents see different plugin sets, which makes
  divergent bugs impossible to debug, and they can mask real config breakage.
  Testing happens through plain `eve tui` against the **configured** plugin
  sources (see "How to test" below). The proper fix for local development is
  local-folder plugin sources (Bug B), not env vars.
- **`eve tui` must work as-is.** No wrapper, no extra flags. If something only
  works with extra parameters, that's a bug to fix, not a workflow to adopt.
- **Do not merge** v4.5 (eve#55 + eve-providers#10, a coupled merge — eve first)
  until *every* bug surfaced during minimalistic usage is fixed AND the code has
  been manually reviewed. The human decides when that bar is met.
- **Every bug gets an entry below** before/while it is fixed, with the repo + PR
  commit that addresses it.

## How to test (`eve tui`, no env vars)

`eve` core runs from this checkout, so it already reflects the `v4.5` branch.
Only the providers/packages are synced from a remote — point that source at the
local checkout (a first-class supported source kind; no remote, no push needed):

```sh
# one-time: make eve use the local v4.5 providers instead of GitHub main
eve plugin source add ../eve-providers --id eve-providers --ref v4.5 --auth none
eve pull                 # materialize; re-run after each agent commit to providers
eve tui                  # plain — no env vars
```

To revert to GitHub: `eve plugin source add https://github.com/fruwehq/eve-providers.git --id eve-providers --ref main --auth none && eve pull`.

### A — tf-stage: read path didn't stage externalized stacks ☑
`status`/`ip`/`start`/`stop`/`password` ran terramate without vendoring the
externalized provider stacks (only the write path did via `tf-init`). On a fresh
tree terramate couldn't discover the stacks → status "not created" and a
terramate error box captured verbatim as the IP. Fixed with a shared core helper
(`scripts/tf-stage` / `stage_provider_stacks`) called from `eve_load_profile_env`
(gated on the terraform engine); `prepare_provider_stacks` is now idempotent so
frequent status polls don't churn the filesystem.
- eve: `fix(tf-dispatch): stage provider stacks on the terraform read path` (eve#55)
- eve-providers: `fix(providers): stage terraform stacks in eve_load_profile_env` (eve-providers#10)

## Architecture workstream (the unified flow)

v4.5 is "make eve usable + fix everything milestone 4 broke." The provider-status
bug below exposed that there is **no single flow** for config/state — each plugin
invents its own notion of "configured," and status vs dispatch resolve config
differently. The fix is structural: one config/secret flow and one state-machine
platform, used by everything.

### M — Status ≡ dispatch: one config/secret resolution ☑(partial)
`scripts/providers-status` ran connectivity with a **bare environment** (no
secrets), so Vultr (gate = `VULTR_API_KEY`, secrets-only) showed ✗ while dispatch
could still create instances; AWS showed "configured" off **ambient `~/.aws`**,
bypassing eve entirely. Immediate fix landed: the probe now resolves the same
`ConfigEnv.environment()` + `_inject_secrets` that dispatch uses (vultr → yes/yes;
aws now honestly reflects eve config). **Systemic (open):** make that resolution
**one shared helper** that is the *only* way any code path gets a provider/
package/instance's config+secrets; no plugin may read ambient external config
behind eve's back. Folds into N/O.
- eve: `fix(providers-status): resolve config-env + secrets like dispatch`

### N — Central state-machine platform ☐
Today state logic is scattered: `state.py` (enums + validation), `state_machine.py`
(derivation helpers), and `desired_state_for`/`provider_state_for`/`record_*`
across dispatch/provider_command/package_dispatch. Consolidate into **one core FSM
platform** behind a clean interface — states, transitions (guards/actions), and a
**context of state variables**; pluggable host-vs-client storage. **Providers,
instances, packages, OS, and bundles are all machines** managed by it; none
reimplement state. Interface kept clean enough to later swap the implementation
for a cns-based FSM (ref: github.com/fruwe/cns_statemachine). Provider
"configured/reachable" becomes an FSM-computed state (not a separate probe).

### O — Settings are FSM context, declared per plugin ☐
Each plugin declares its config params, defaults, and an explicit **unset** state
(empty string only where sufficient). Resolution order: plugin default → global
override (settings menu / CLI / yaml) → **per-instance override** (instance
settings / CLI / yaml), for ALL config (not just packages). The FSM platform
resolves and **provides settings as machine context** — so status, dispatch, and
connectivity always see identical config. This is what makes M structurally
impossible to regress.

### P — Config → XDG, user-global ☐
Move off project-local `.eve/`: `~/.config/eve/` (config.yaml, secrets/),
`~/.local/state/eve/` (instances/, state/), `~/.cache/eve/` (sources/ plugin
cache), `~/.local/share/eve/` (plugins/). Include a migration from an existing
`.eve/`. eve becomes a user-level tool, not per-checkout.

### Q — TUI is a view over the CLI surface ☐
Everything visible/doable in the TUI must be reproducible as a CLI command — one
shared command surface, with 1:1 parity enforced by tests. (Warm in-process engine
is fine for speed; the invariant is parity, not literally shelling out.)

### R — CLI streaming ≡ non-streaming ☐
No command or option may exist in only streaming or only non-streaming mode;
enforce parity with a test.
