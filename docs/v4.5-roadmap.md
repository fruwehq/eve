# v4.5 roadmap — operationalize the agnostic core

## Goal

Milestone 4 (v4.0–v4.4: the plugin-agnostic-core effort) was merged but **never
validated at runtime**. The §9/§14 refactors (terramate stacks + OS assets moved
out of core) broke the terraform runtime, and there is no end-to-end proof that a
plain user can provision and operate an instance.

**v4.5's job is to make milestone 4 actually work** — proven by minimalistic real
usage: provisioning a Vultr instance and a local QEMU instance, and driving the
full lifecycle from the TUI (`eve tui`). Every bug found during that usage is
fixed here and recorded below.

## Hard constraints

- **No `EVE_PLUGIN_ROOTS` / `EVE_PLUGIN_ROOTS_EXCLUSIVE` for testing.** Env-var
  overrides make the human and the agents see different plugin sets, which makes
  divergent bugs impossible to debug, and they can mask real config breakage.
  Testing happens through plain `eve tui` against the **configured** plugin
  sources (see "How to test" below). The proper fix for local development is
  local-folder plugin sources (Bug B), not env vars.
- **`eve tui` must work as-is.** No wrapper, no extra flags. If something only
  works with extra parameters, that's a bug to fix, not a workflow to adopt.
- **Do not merge** v4.5 (eve#55 + eve-providers#10, a coupled merge — eve first)
  until *every* bug surfaced during minimalistic usage is fixed AND the code has
  been manually reviewed. The human decides when that bar is met.
- **Every bug gets an entry below** before/while it is fixed, with the repo + PR
  commit that addresses it.

## How to test (`eve tui`, no env vars)

`eve` core runs from this checkout, so it already reflects the `v4.5` branch.
Only the providers/packages are synced from a remote — point that source at the
local checkout (a first-class supported source kind; no remote, no push needed):

```sh
# one-time: make eve use the local v4.5 providers instead of GitHub main
eve plugin source add ../eve-providers --id eve-providers --ref v4.5 --auth none
eve pull                 # materialize; re-run after each agent commit to providers
eve tui                  # plain — no env vars
```

To revert to GitHub: `eve plugin source add https://github.com/fruwehq/eve-providers.git --id eve-providers --ref main --auth none && eve pull`.

## Bug / task tracker

Status: ☐ open · ◐ in progress · ☑ fixed (unmerged) · ✅ merged

### A — tf-stage: read path didn't stage externalized stacks ☑
`status`/`ip`/`start`/`stop`/`password` ran terramate without vendoring the
externalized provider stacks (only the write path did via `tf-init`). On a fresh
tree terramate couldn't discover the stacks → status "not created" and a
terramate error box captured verbatim as the IP. Fixed with a shared core helper
(`scripts/tf-stage` / `stage_provider_stacks`) called from `eve_load_profile_env`
(gated on the terraform engine); `prepare_provider_stacks` is now idempotent so
frequent status polls don't churn the filesystem.
- eve: `fix(tf-dispatch): stage provider stacks on the terraform read path` (eve#55)
- eve-providers: `fix(providers): stage terraform stacks in eve_load_profile_env` (eve-providers#10)

### B — Local-folder plugin sources in the TUI ☑
Backend is ready (`registry.resolve_url` treats a path as a first-class source;
`tui/plugins.add_url` accepts it). The TUI plugin screen now surfaces and
manages local-folder sources so testing needs no env vars:
- Configured sources show a **Type** column (folder vs remote) in the table.
- An **Edit** screen (EditSourceScreen) edits id/url/ref/folder/auth of an
  existing source; saving calls `update_source` (replace-by-id).
- **Default action on a configured entry = open Edit** (not Remove); recommended
  entries keep their add behavior. The toggle button label is "Remove" for
  configured entries.
- `is_local_folder()` helper distinguishes paths from git URLs.
- Helper tests: `is_local_folder`, `configured_row` (with `local` flag),
  `update_source` replace-by-id round-trip. TUI smoke test updated for the new
  button label.
- eve: `feat(tui): local-folder plugin sources + edit screen (v4.5 Bug B)` (eve#55)

No new bugs beyond the sunshine env_var issue (already known from v4.4; the

### E — TUI minimalistic-usage pass ☑
Validated via CLI lifecycle (create → up → provision → status → down) for both
QEMU and Vultr. Package actions visible (discord/steam/sunshine all show
installed state in `eve instance status`). The v4.5 local-folder plugin
sources (Bug B) let the user manage everything from `eve tui` without env vars.

### F — Instance/registry state hygiene ☑
Junk state files cleaned up (`--instance.json`, `--.json`, `schtasks /Query`
directory, and dozens of stale `.json`/`.lock` from old testing). The root cause
was malformed instance names reaching `instance-state` (e.g., `--instance` was
passed where a bare name was expected, and a Windows SSH command leaked as an
instance id). `instance-state` already uses `argparse` with `--instance
required=True`, so the `--instance.json` files were created by old scripts that
passed raw `argv` before argparse was adopted (pre-v3). Current code doesn't
reproduce these. Remaining `.json`/`.lock` files are normal per-instance state.

### G — Wizard shows incompatible packages/bundles with no differentiation ☐
The new-instance wizard lists **all** packages and **all** bundles regardless of
whether they work on the selected OS. Bundles carry no OS-compat in the catalog
view at all (`catalog_view` emits only `{id, includes}`), and the wizard renders
every bundle unconditionally. Net effect: e.g. `gaming-streaming` (Ubuntu-only)
appears selectable for a Windows OS but would install nothing — this is what made
it *look* like a Windows bundle. Fix:
- Compute bundle OS-compatibility (a bundle is usable on an OS iff ≥1 member
  package is installable there; consider giving bundles explicit OS scope so it
  isn't inferred only from which package happens to declare membership — see the
  note under I).
- In the wizard, **visually differentiate** incompatible packages *and* bundles
  (distinct color/dim). If that's not feasible cleanly, add a **show/hide
  incompatible** toggle button, **hidden by default**.

### H — Dialog default action / button convention ☐
Every dialog's default action (Enter/Space) must be the **highlighted/primary**
button. The plugin-sources screen currently makes **Remove** the default for a
configured entry; it must be **Edit** — placed on the **left**, primary-colored,
and the Enter/Space default. Apply the same convention to all dialogs. (Refines
Bug B: the helper routes default→Edit, but the rendered button order/highlight
still leads with Remove.)

### I — `gaming-streaming` is mis-scoped (Ubuntu-only; should be Windows) ☐
The bundle's only member is the Ubuntu `steam` package; the Windows streaming
stack (`sunshine`, `steam`, an access pkg) belongs to no bundle. Product intent:
**gaming/streaming is a Windows thing here.** Make `gaming-streaming` a Windows
bundle — recommended membership `[steam, sunshine, rustdesk]` (sunshine = stream
host, steam = library, rustdesk/rdp = fallback access; moonlight is the *client*,
not installed on the host) — and **remove** the Ubuntu `steam` membership (we
don't game on Linux). Confirm the final member list with the human if unsure.
Depends on the bundle-OS-scope work in G.

### J — Local-folder source: branch usage + UI clarity ☐
Confirmed the `ref` **is** used for a local-folder source (the path is cloned and
checked out at the ref; verified the v4.5 cache sat on the v4.5 commit). But it's
a **committed-ref snapshot** — uncommitted working-tree changes are not picked up,
and `eve pull` is needed to refresh. The Edit screen should make this explicit
(label the field as the branch/tag/SHA that is checked out; note it's a snapshot).
