# v3.3 Roadmap — Ubuntu host + Docker container deployability

Companion to [v3.1-roadmap.md](v3.1-roadmap.md) and
[v3.2-roadmap.md](v3.2-roadmap.md).

Today Eve runs on macOS (the only validated host). The codebase
implicitly assumes:

- macOS-specific tools available (`dscacheutil` for DNS, BSD `sed`,
  Homebrew binary paths, `dscacheutil`, `osascript` for some prompts).
- Local provider chain assumes `vagrant-qemu` plus a Homebrew
  `qemu-system-aarch64` install.
- The user runs `make` from a checkout on their workstation.

v3.3 broadens this so Eve runs fully on:

1. **Ubuntu (host machine)** — A user with an Ubuntu workstation can
   `git clone`, `make doctor`, and run the same workflows as on
   macOS: create instances, provision, TUI, smoke tests.
2. **Docker (full runtime)** — A single `docker run eve/eve:latest`
   (or `docker compose up`) brings up Eve with the TUI, the
   instance registry, and all provider tooling, persisting state to
   a mounted volume.

The two targets share most of the work (de-mac-ifying the codebase),
then diverge into Linux-host vs container-specific concerns.

## Goals

1. **Ubuntu host parity.** Every `make` target that works on macOS
   works on Ubuntu, with the same UX.
2. **Reproducible Docker image.** Eve ships a Docker image that bundles
   Ruby, Python (Textual), Terramate/Terraform, vagrant + qemu (when
   relevant), gcloud/aws/vultr CLIs, and the runtime env.
3. **Persistent state.** Both targets respect a writable `EVE_HOME`
   (or equivalent) so the instance registry, generated artifacts,
   and provider state survive container restarts / host reboots.
4. **Clear deployment recipes.** README has copy-paste sections for
   both: "Run Eve on Ubuntu" and "Run Eve in Docker".

## Audit checklist

Things that currently assume macOS:

| Area | macOS today | Needs |
|---|---|---|
| DNS resolution | `dscacheutil -q host -a name` in raspberry-pi/commands/ip | Linux fallback (`getent ahostsv4`) — already present, just needs Linux verification |
| `sed -i` syntax | BSD `sed -i.bak` | Linux `sed -i` — already mostly handled |
| Realpath / symlink | `realpath -f` differs | use `readlink -f` or `coreutils realpath` |
| Homebrew binary paths | hardcoded paths in some tool checks | use `command -v` everywhere |
| `osascript` prompts | TUI/eve helpers | dialog (Linux), zenity, or skip |
| Vagrant + QEMU on Linux | tested on macOS only | document libvirt alternative; verify vagrant-qemu plugin on Linux |
| `make doctor` | tool list is macOS-flavored | add Linux equivalents |
| pwd/realpath in scripts | mostly portable | spot-check |
| `find ... -name` regex order | already alphabetized (per AGENTS.md) | verify |
| `mktemp -d ${TMPDIR:-/tmp}` | works both | ✅ |
| File watch / TUI rendering | Textual is cross-platform | ✅ |

## Docker image design

### Base image

`ubuntu:24.04` (matches the OS Eve provisions on guests; LTS).

### Contents

```
/opt/eve/                       # repo, mounted as read-only or built-in
/data/                          # mountable volume:
  /data/.eve/                   # instance registry, state, secrets
  /data/.generated/             # per-instance workdirs
  /data/.env.local              # optional user env file
```

Volume mount: `-v "$HOME/.eve-data:/data"` (host) → `EVE_HOME=/data`
(container).

### Dependencies bundled

- Ruby 3.4 + `json_schemer`, `bundler`
- Python 3.13 + Textual + Eve TUI deps via `poetry install --no-dev`
- Terraform + Terramate
- Provider CLIs: `aws`, `gcloud`, `vultr-cli` (selectable via image
  variant if size matters)
- `vagrant` + `qemu-system-*` if local-qemu in container is desired
  (separate larger image variant; default image omits it because
  qemu-in-docker on macOS host is fiddly).
- SSH client + `jq` + `yq` + `curl`

### Image variants

- `eve/eve:latest` — slim, no qemu, no vagrant. Post-trim v3.3 target:
  under 1.6 GB uncompressed on the CI runner and under 500 MB as a
  compressed pull/archive. Use for cloud providers (AWS/GCP/Vultr/TrueNAS)
  and remote Raspberry Pi.
- `eve/eve:full` — adds vagrant + qemu for local-qemu use. ~1.5 GB.

### Build

`docker/Dockerfile` already exists for the optional toolchain; v3.3
either extends it or replaces with a multi-stage build that produces
both the dev toolchain (`make docker.test`) and the runtime image
(`docker run eve`).

### TUI in a container

Textual works in any TTY. `docker run -it eve/eve` brings up the
TUI. SSH/X11 forwarding for the spawned `make ssh` is the only
container-specific concern; either inherit the user's SSH agent
(`-v ~/.ssh:/root/.ssh:ro -v $SSH_AUTH_SOCK:/ssh-agent
-e SSH_AUTH_SOCK=/ssh-agent`) or document the SSH-from-host
alternative.

## Ubuntu host

### Setup recipe (target end state)

```bash
# 1. System deps
sudo apt install ruby ruby-dev terraform jq yq sshpass python3-poetry

# 2. Terramate
curl -fsSL https://github.com/terramate-io/terramate/releases/latest/...

# 3. Eve
git clone <repo>
cd ephemeral-cloud-gaming
poetry install
make doctor   # should report all green on Ubuntu

# 4. First-run config (when v3.2 lands)
make eve
```

### Known gaps

- `vagrant-qemu` plugin works on Linux but the catalog default
  `vagrant_box` may be an arm64 macOS-friendly box; document
  alternates.
- `dscacheutil` doesn't exist on Linux — already has a `getent`
  fallback, just needs Linux smoke.
- `pwsh` for Windows guests' provisioning is not Linux-relevant; the
  Eve HOST doesn't need pwsh, just `ssh`.

### `make doctor` updates

Today's `make doctor` is a flat list. v3.3 makes it OS-aware: same
list, but each entry knows the OS-specific install command suggestion
(`brew install X` on macOS, `apt install X` on Ubuntu).

## Implementation order

1. **Audit.** Run `scripts/test-shellcheck` on Ubuntu; capture every
   incompatibility.
2. **De-mac-ify scripts.** Replace `dscacheutil`, BSD `sed`, etc.
3. **Ubuntu smoke.** Get the full test suite green on Ubuntu CI.
4. **Dockerfile.** Multi-stage build for `eve/eve:latest` (slim) and
   `eve/eve:full` (with qemu).
5. **Volume contract.** Document and enforce `EVE_HOME=/data`.
6. **docker compose** recipe for `up`/`down` from a single yaml.
7. **README sections** for both runtimes.
8. **CI matrix.** Add Ubuntu to the existing test matrix.

## Non-goals

- Windows as a host. (You can run via WSL2 + Docker Desktop today.)
- Replacing Vagrant with a different local VM runner.
- Multi-tenant SaaS Eve. Eve is single-user by design.

## Acceptance

- `make test` is green on a fresh Ubuntu 24.04 host with `apt`-only deps.
- `docker run -it -v ~/.eve-data:/data eve/eve:latest make eve`
  opens the TUI and can drive a cloud provider end-to-end.
- README has a "Quickstart on Ubuntu" and "Quickstart with Docker"
  section, each with copy-pastable commands.
- CI runs the test suite on both macOS and Ubuntu.
