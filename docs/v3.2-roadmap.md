# v3.2 Roadmap — Structured configuration + TUI config screens

Companion to [v3.1-roadmap.md](v3.1-roadmap.md).

v3.1 introduced `.eve/config.yaml` as a structured config alongside
`.env` and `.env.local`, but the migration is incomplete: `.env` is
still the canonical place for runtime settings, secrets, provider
credentials, and per-provider options. There is no in-TUI editing.

v3.2 finishes the migration so that:

- **Non-secret configuration lives in `.eve/config.yaml`** (structured,
  documented, validated).
- **Secrets live in a clear secret store** (`.env.local` keeps working
  as a fallback, but secrets are scoped per provider where possible).
- **The TUI has a Settings screen** for global options
  (display resolution, default location, VM user name, refresh
  intervals, etc.).
- **Each provider has its own Configuration screen** in the TUI for
  provider-specific settings (AWS profile/region, GCP project, Vultr
  API key, TrueNAS host/SSH key, Raspberry Pi host/IP, etc.).
- **`.env` becomes optional** — Eve runs fine without it as long as
  `.eve/config.yaml` and the secret store are populated.

## Goals

1. **Single source of truth per setting.** A user can ask "where is X
   configured?" and the answer is one file, in one place, with one
   precedence rule.
2. **Discoverability via the TUI.** No more reading `.env` to find
   what to set; the Settings and Provider Configuration screens list
   every option, its current value, source, and how to change it.
3. **Schema-validated.** Each setting has a type, a default, an
   optional secret marker. Invalid configs fail loudly at load time.
4. **Backwards-compatible.** Existing `.env` and `.env.local` keep
   working; `make config.migrate` (already in tree) handles non-secret
   values, and v3.2 finishes the per-provider scope.

## Design

### Configuration layers (final precedence, top wins)

1. Command-line overrides (`make foo VAR=x`)
2. Per-instance registry overrides (`.eve/instances.yaml`)
3. `.eve/config.yaml` (non-secret, TUI-editable)
4. Provider-scoped secret store (`.eve/secrets/<provider>.yaml`, mode 0600)
5. `.env.local` (legacy fallback, still works)
6. `.env` (legacy fallback)
7. `config/defaults.yaml` (committed defaults)
8. Catalog defaults (`config/catalog.yaml` machine `defaults:` block)

v3.1 already implements layers 1-3, 5-8. v3.2 adds layer 4 and the
TUI editing surface.

### Schema

Extend `core/schema/` with:

- `config.schema.json` — global config shape, mirrors the structure of
  `config/defaults.yaml`.
- `provider-config.schema.json` — per-provider scoped config. Each
  provider plugin's manifest declares its config schema fragment under
  a new top-level `config_schema:` field; the resolver merges them.

### Secret store

`.eve/secrets/<provider>.yaml` (mode 0600, gitignored). Format:

```yaml
aws:
  access_key_id: <secret>
  secret_access_key: <secret>
  profile: <non-secret, may live in config.yaml instead>
```

Secrets never appear in TUI plaintext display (masked). Secret values
flow into the resolved instance JSON via the same env_or() path
already used by `scripts/profile-resolve`.

### TUI Settings screen

Reachable from the main TUI via `S` keybind or the menu. One pane per
section, mirroring `config/defaults.yaml`:

- Global: vm_user_name, my_ip, ssh_public_key_file, timezone
- Display: resolution, fps
- Moonlight: bitrate, video_codec, video_decoder, display_mode
- Sunshine: max_bitrate_kbps, version
- RustDesk: server, password (masked)
- ThinLinc: agent_hostname, accept_eula, etc.

Each field shows: current value, source (config.yaml / env / default),
and an Edit button. Edits write to `.eve/config.yaml`; secrets prompt
for the secret store.

### TUI Provider Configuration screen

Reachable from the provider pane (added by v3.1 finish work). One
screen per provider, listing the fields declared by that provider's
`config_schema:` manifest entry. Example for AWS:

```
AWS Configuration
─────────────────
profile:                  default              [config.yaml]
region:                   us-east-1            [config.yaml]
config_file:              ~/.aws/config        [env]
shared_credentials_file:  ~/.aws/credentials   [env]
access_key_id:            ●●●●●●●● (masked)    [secret store]
secret_access_key:        ●●●●●●●● (masked)    [secret store]

[Test connection]  [Save]  [Cancel]
```

The `Test connection` button dispatches the provider's existing
`status` or a new `validate-config` action.

## Implementation order

1. **Schema.** Author `config.schema.json` and the manifest's new
   `config_schema:` field; validate every load.
2. **Secret store.** Implement `.eve/secrets/<provider>.yaml`
   read/write through `core/sdk/secrets.rb` (file lock + 0600 mode);
   wire into `scripts/profile-resolve`'s env emission.
3. **TUI Settings screen.** Static fields from `config/defaults.yaml`
   shape.
4. **TUI Provider Configuration screen.** Dynamic fields driven by
   each provider's `config_schema:`.
5. **Migration helper.** Extend `scripts/config-migrate` to cover the
   per-provider secret split.
6. **Documentation + golden test updates.**

## Non-goals

- A web UI. The TUI is the canonical interface.
- Replacing `config/catalog.yaml`. The catalog stays committed; only
  user-specific config moves.
- Backwards-incompatible changes to `.env`. Users who prefer it keep
  using it; v3.2 just doesn't require it anymore.

## Acceptance

A user can:

1. Clone the repo with **no `.env` file**, run `make eve`, and the
   TUI walks them through a one-time Settings + first-provider
   configuration before any instance can be created.
2. Edit provider credentials via the TUI without touching any file
   on disk by hand.
3. See exactly where each setting comes from in the TUI (config.yaml,
   secret store, env, or default).
4. Run the existing test suite green without any `.env` file present.
