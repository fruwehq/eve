# v3.2 Roadmap — Structured configuration + TUI config screens

Companion to [v3.1-roadmap.md](v3.1-roadmap.md).

v3.1 introduced `.eve/config.yaml` as a structured config alongside
`.env` and `.env.local`, but the migration is incomplete: `.env` is
still the canonical place for runtime settings, secrets, provider
credentials, and per-provider options. There is no in-TUI editing.

v3.2 finishes the migration so that:

- **Non-secret configuration lives in `.eve/config.yaml`** (structured,
  documented, validated).
- **Secrets live in a clear secret store** (`.env.local` keeps working
  as a fallback, but secrets are scoped per provider where possible).
- **The TUI has a Settings screen** for global options
  (display resolution, default location, VM user name, refresh
  intervals, etc.).
- **Each provider has its own Configuration screen** in the TUI for
  provider-specific settings (AWS profile/region, GCP project, Vultr
  API key, TrueNAS host/SSH key, Raspberry Pi host/IP, etc.).
- **`.env` becomes optional** — Eve runs fine without it as long as
  `.eve/config.yaml` and the secret store are populated.

## Goals

1. **Single source of truth per setting.** A user can ask "where is X
   configured?" and the answer is one file, in one place, with one
   precedence rule.
2. **Discoverability via the TUI.** No more reading `.env` to find
   what to set; the Settings and Provider Configuration screens list
   every option, its current value, source, and how to change it.
3. **Schema-validated.** Each setting has a type, a default, an
   optional secret marker. Invalid configs fail loudly at load time.
4. **Backwards-compatible.** Existing `.env` and `.env.local` keep
   working; `make config.migrate` (already in tree) handles non-secret
   values, and v3.2 finishes the per-provider scope.

## Design

### Configuration layers (final precedence, top wins)

1. Command-line overrides (`make foo VAR=x`)
2. Per-instance registry overrides (`.eve/instances.yaml`)
3. `.eve/config.yaml` (non-secret, TUI-editable)
4. Provider-scoped secret store (`.eve/secrets/<provider>.yaml`, mode 0600)
5. `.env.local` (legacy fallback, still works)
6. `.env` (legacy fallback)
7. `config/defaults.yaml` (committed defaults)
8. Catalog defaults (`config/catalog.yaml` machine `defaults:` block)

v3.1 already implements layers 1-3, 5-8. v3.2 adds layer 4 and the
TUI editing surface.

### Schema

Extend `core/schema/` with:

- `config.schema.json` — global config shape, mirrors the structure of
  `config/defaults.yaml`.
- `provider-config.schema.json` — per-provider scoped config. Each
  provider plugin's manifest declares its config schema fragment under
  a new top-level `config_schema:` field; the resolver merges them.

### Secret store

`.eve/secrets/<provider>.yaml` (mode 0600, gitignored). Format:

```yaml
aws:
  access_key_id: <secret>
  secret_access_key: <secret>
  profile: <non-secret, may live in config.yaml instead>
```

Secrets never appear in TUI plaintext display (masked). Secret values
flow into the resolved instance JSON via the same env_or() path
already used by `scripts/profile-resolve`.

### TUI Settings screen

Reachable from the main TUI via `S` keybind or the menu. One pane per
section, mirroring `config/defaults.yaml`:

- Global: vm_user_name, my_ip, ssh_public_key_file, timezone
- Display: resolution, fps
- Moonlight: bitrate, video_codec, video_decoder, display_mode
- Sunshine: max_bitrate_kbps, version
- RustDesk: server, password (masked)
- ThinLinc: agent_hostname, accept_eula, etc.

Each field shows: current value, source (config.yaml / env / default),
and an Edit button. Edits write to `.eve/config.yaml`; secrets prompt
for the secret store.

### TUI Provider Configuration screen

Reachable from the provider pane (added by v3.1 finish work). One
screen per provider, listing the fields declared by that provider's
`config_schema:` manifest entry. Example for AWS:

```
AWS Configuration
─────────────────
profile:                  default              [config.yaml]
region:                   us-east-1            [config.yaml]
config_file:              ~/.aws/config        [env]
shared_credentials_file:  ~/.aws/credentials   [env]
access_key_id:            ●●●●●●●● (masked)    [secret store]
secret_access_key:        ●●●●●●●● (masked)    [secret store]

[Test connection]  [Save]  [Cancel]
```

The `Test connection` button dispatches the provider's existing
`status` or a new `validate-config` action.

## Implementation order

1. **Schema.** Author `config.schema.json` and the manifest's new
   `config_schema:` field; validate every load.
2. **Secret store.** Implement `.eve/secrets/<provider>.yaml`
   read/write through `core/sdk/secrets.rb` (file lock + 0600 mode);
   wire into `scripts/profile-resolve`'s env emission.
3. **TUI Settings screen.** Static fields from `config/defaults.yaml`
   shape.
4. **TUI Provider Configuration screen.** Dynamic fields driven by
   each provider's `config_schema:`.
5. **Migration helper.** Extend `scripts/config-migrate` to cover the
   per-provider secret split.
6. **Documentation + golden test updates.**

## Non-goals

- A web UI. The TUI is the canonical interface.
- Replacing `config/catalog.yaml`. The catalog stays committed; only
  user-specific config moves.
- Backwards-incompatible changes to `.env`. Users who prefer it keep
  using it; v3.2 just doesn't require it anymore.

## Acceptance

A user can:

1. Clone the repo with **no `.env` file**, run `make eve`, and the
   TUI walks them through a one-time Settings + first-provider
   configuration before any instance can be created.
2. Edit provider credentials via the TUI without touching any file
   on disk by hand.
3. See exactly where each setting comes from in the TUI (config.yaml,
   secret store, env, or default).
4. Run the existing test suite green without any `.env` file present.

## `core/sdk/secrets.rb` API

Mirror of `core/sdk/state.rb` shape (file-locked accessor) but
scoped per provider. Single point of read/write for everything in
`.eve/secrets/<provider>.yaml`.

```ruby
module Eve
  module SDK
    module Secrets
      class SecretsError < StandardError; end

      # File path for a provider's secret YAML. Honors EVE_SECRETS_DIR
      # override (mirrors EVE_STATE_DIR for tests).
      def self.path_for(provider_id) end

      # Read the provider's secrets hash; returns {} if file absent.
      # Raises SecretsError on parse error or wrong file mode.
      def self.read(provider_id) end

      # Atomic write: tempfile + rename + chmod 0600 + fsync.
      # Raises SecretsError if any secret value is not a String.
      def self.write(provider_id, hash) end

      # Merge given hash into existing; useful for TUI partial edits.
      # Atomic via with_lock; pass nil values to unset a key.
      def self.update(provider_id, partial) end

      # Delete a single secret (key path) or the whole file when keys
      # is the literal :all symbol.
      def self.delete(provider_id, keys: :all) end

      # Yield a writable hash inside a flock-exclusive critical section.
      def self.modify(provider_id) end

      # Return value or nil; never throws on missing file or missing key.
      def self.get(provider_id, key) end
    end
  end
end
```

Storage format (mode 0600, gitignored):

```yaml
# .eve/secrets/aws.yaml
aws:
  access_key_id: "<value>"
  secret_access_key: "<value>"
```

Top-level key matches the provider id so consumers grep'd
`aws.access_key_id` resolve unambiguously. Multi-provider secrets
(e.g. a shared SSH key) get one secret entry per provider that
references them; we do not de-dupe.

### Wire into the resolver

`scripts/profile-resolve --emit env` already calls `env_or(NAME;
default)` for each emitted env var. v3.2 adds a step before that
which:

1. Reads `.eve/secrets/<provider>.yaml` for the resolved instance's
   provider.
2. Populates a process-scoped hash, preferring `ENV[NAME]` when set
   (env wins over secrets store; matches the precedence table).
3. Exports the merged env block so `env_or()` sees the secrets store
   values as if they came from `.env.local`.

No change to the env names themselves; the secrets store is an
additional source for the same names.

## `config_schema:` manifest field shape

Provider plugins gain an optional top-level `config_schema:` field
declaring their config + secret fields. Schema enforced by an
addition to `core/schema/plugin-manifest.schema.json`.

```yaml
# plugins/providers/aws/eve-plugin.yaml (excerpt)
config_schema:
  config:
    profile:
      type: string
      description: "AWS named profile to use"
    region:
      type: string
      enum: [us-east-1, us-west-2, eu-west-1, ap-northeast-1, ...]
    config_file:
      type: path
      default: "~/.aws/config"
    shared_credentials_file:
      type: path
      default: "~/.aws/credentials"
  secrets:
    access_key_id:
      type: string
      description: "AWS_ACCESS_KEY_ID"
    secret_access_key:
      type: string
      description: "AWS_SECRET_ACCESS_KEY"
```

Each field gains: `type` (string/integer/boolean/path/url/host/enum),
optional `enum:`, optional `default:`, optional `description:`,
optional `required: true` (for "must be set before any provider
command works").

TUI renders the screen from this schema. The provider's `status`
command stays as the "validate connection" mechanism — no new
schema field needed.

## TUI interaction model

### Settings screen

- Keybind: `S` (global, when no modal is open).
- Layout: single screen, scrollable list grouped by section. Each
  row: `<label> <value> <source-tag>`. Sources: `[config.yaml]`,
  `[secrets]`, `[env]`, `[default]`.
- Edit: focus a row, press `Enter` → modal `EditFieldScreen` with a
  text input pre-filled with current value. Secret fields show a
  masked input. `Esc` cancels, `Enter` saves.
- Save: write to `.eve/config.yaml` (or the secrets store) via the
  SDK, atomic write. Reload the screen so source tags update.
- Reset: row context menu (`R`) clears the override; field falls
  back to next layer in the precedence chain.

### Provider Configuration screen

- Reachable from the ProviderPane (added in v3.1 Part 3) via a
  `Configure` button per provider row.
- Same edit/save/reset shape as Settings, but populated from the
  provider's `config_schema:`.
- `Test connection` button at the bottom dispatches `make
  provider.action PROVIDER=<id> ACTION=status` and surfaces the
  result inline.

### First-run flow (cold checkout, no `.env`)

When `make eve` starts and `Eve::SDK::Resolve.from_env` would fail
because required fields are missing, the TUI shows a `FirstRunScreen`
instead of the main view:

1. Welcome message + summary of what's missing.
2. Buttons: `Open Settings`, `Configure Provider <id>` (one per
   provider that declared `required:` fields), `Skip for now`.
3. After all required fields are set, the main view loads.

## Phase split

Each phase is a self-contained PR. Each unblocks the next.

### Phase 1 — Schemas + manifests (start here)

- `core/schema/config.schema.json` mirroring `config/defaults.yaml`
  shape; validates a future `.eve/config.yaml`.
- Extend `core/schema/plugin-manifest.schema.json` to allow optional
  `config_schema:` on provider manifests, with required `config:`
  and `secrets:` sub-keys (both maps from field-name to field-spec).
- Add `config_schema:` to every provider manifest using the
  inventory above.
- `make test.schemas` validates `config/defaults.yaml` against the
  new config schema and every provider manifest's `config_schema:`.

### Phase 2 — Secret store

- `core/sdk/secrets.rb` with the API above.
- `scripts/profile-resolve` integration: read provider's secrets
  before emitting env.
- `make test` updated with parallel-writer + missing-file +
  mode-not-0600 tests (mirror state-concurrency suite).
- `scripts/config-migrate` extended: when run on a `.env.local` that
  contains a known secret env name (e.g. `VULTR_API_KEY`), move it
  into the matching secrets file and comment out the env line.

### Phase 3 — TUI Settings screen

- `tui/widgets/settings_screen.py` (or extend `tui/widgets.py`).
- Sections sourced from `config/defaults.yaml`; per-field rendering
  driven by `config.schema.json`.
- `EditFieldScreen` modal for editing values.
- Wired into the global keymap (`S`).
- pytest coverage for the pure helpers (source-resolver, field-renderer)
  in `tests/python/test_settings_screen.py`.

### Phase 4 — TUI Provider Configuration screen

- New screen class driven by provider manifest's `config_schema:`.
- Reachable from ProviderPane (existing v3.1 widget) with a
  `Configure` button per provider row.
- `Test connection` dispatches `provider.action status`.

### Phase 5 — First-run flow + polish

- `FirstRunScreen` when required fields missing.
- `make doctor` augmented to show which required fields are
  populated.
- README updates documenting the new path; `.env` becomes a legacy
  fallback in docs.

## Implementation notes

- Top-level `secrets:` and `config:` under `config_schema:` keeps the
  two concerns distinct, lets the TUI render different styling for
  secret fields, and lets the file routing rule be a one-liner.
- The secrets file does NOT inherit from `.env.local`; it's a clean
  new home. `scripts/config-migrate` is the bridge for existing
  installs.
- The provider plugin owns its `config_schema:`. External plugins
  (hello-package, echo-provider, the fake provider) can declare
  their own and the TUI renders them with no core changes — same
  pattern as `actions:`.
- The TUI never displays a secret in plaintext, even when "editing"
  — the input field receives the new value but never reads the old
  one back from disk.
