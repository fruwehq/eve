# v4.0 Roadmap â€” Agnostic plugin host, external registry, and a real `eve` CLI

Companion to [v3.5-roadmap.md](v3.5-roadmap.md). Where v3.1â€“v3.5 hardened the
*plugin contract* and orchestration internals, v4.0 turns `eve` into a
**provider/package-agnostic host**: the core ships **no** providers, **no**
packages, and **no** catalog data. Everything that today lives in `plugins/`
and `config/catalog.yaml` is pulled at runtime from **external git repositories**
(including private GitHub), version-pinned and independently testable.

## Context â€” the problem

Today the core is entangled with its content in three places:

1. **`config/catalog.yaml`** is a central, hand-maintained registry of every
   machine, OS, init, package, and bundle. Adding a provider or package means
   editing core config.
2. **Core code has provider-specific branches.** `scripts/catalog-options`
   (`os_available_for_provider`) `match`es on literal provider names; the core
   "knows" about `vultr`, `truenas`, `raspberry-pi`, etc.
3. **Built-in plugins ship in-repo** under `plugins/providers/*` and
   `plugins/packages/*`, co-versioned with the core.

This makes every new domain â€” Windows gaming, OpenWrt routers, AI workstations â€”
land as more clutter inside one repo, tested by one ever-slower suite. The user
goal: keep `eve`'s general purpose (stand up instances and manage their content)
and let **anyone extend it from their own repos**, independently and testably,
without touching core.

The foundations already exist and v4.0 builds on them rather than replacing them:

- `scripts/plugins-sync` (Python) already clones pinned `{id,url,ref}` git
  sources into `.eve/plugins/<id>`, rejecting unpinned refs by default.
- `eve_sdk/plugin_manifest.py` already discovers plugins from **two roots** â€”
  `<repo>/plugins` (builtin) and `Workdir.plugins_dir()` (`.eve/plugins`, synced)
  â€” plus an `EVE_PLUGIN_ROOTS` override, and tags each `_source` as
  `builtin`/`external`.
- Providers already carry `access`, `config_schema` (with `secrets`/`env_var`),
  `capabilities`, `supports`, and their own Terraform stacks in their manifest.
- A fingerprint-keyed cache (`.generated/cache/plugin-list.json`) already
  amortizes manifest parsing for the TUI hot path.

What's missing is: catalog contributed *by plugins* (not central), core with the
provider switches removed, private/subfolder source support, a lockfile, a
conformance test harness, and a real CLI.

## Goals

1. **Agnostic core.** The `eve` repo contains the SDK, CLI, plugin registry,
   contract schemas, instance/state model, and tests â€” and **zero** providers,
   packages, or catalog data. Grep for any provider/OS literal in core returns
   nothing (the `core-boundary` check graduates from "no provider refs in
   `_common`" to "no provider refs anywhere in core").
2. **External, multi-source registry.** Users add any number of git endpoints,
   each scoped to a **subfolder**, including **private GitHub** repos, and
   `eve pull` (manual or scheduled) fetches the newest pinned providers/packages.
3. **Catalog by contribution.** Each provider plugin contributes its machines/
   OSes/inits; each package plugin contributes itself and its bundles. The
   effective catalog is aggregated at load time. `config/catalog.yaml` ceases to
   be authored by hand.
4. **Independent + testable plugins.** A published contract + `eve plugin test`
   conformance harness lets a plugin repo CI itself against the core contract
   with no core checkout.
5. **A real `eve` CLI** (gcloud-style tree) replacing the 480-line / 94-target
   Makefile as the human surface.
6. **Warm execution.** Catalog + plugins parse **once per session**, not once
   per command, so the TUI and scripted pipelines stop paying cold-start per op
   (continuation of v3.5's "fewer, fatter processes").

Non-goal of "agnostic": losing today's behavior. After v4.0, a fresh `eve` plus
`eve pull` of the reference plugin repos reproduces **exactly** today's
functionality (every provider, package, bundle).

## Target architecture

### The plugin contract (core owns the schemas, not the content)

Core publishes versioned JSON schemas under `core/schema/` defining what a plugin
*is*. Extend the existing contract with three additions:

- **Catalog contribution.** A provider manifest gains a `catalog:` block listing
  the `machines`, `oses`, and `inits` it stands up (the rows that live in
  `config/catalog.yaml` today, moved next to the provider that owns them). A
  package manifest may contribute `bundles:`. Core aggregates all contributions
  into the effective catalog; `os_available_for_provider`'s hardcoded `match`
  is replaced by provider-declared OS/image compatibility metadata.
- **Package `config_schema` + artifacts.** Packages get the same
  `config_schema` providers already have (namespaced structured config, e.g.
  `openwrt.wifi_ssid`, validated against the package's schema), plus an
  **artifacts** convention: a package may declare build outputs and a
  `download`-style action that ships an artifact from the instance to the
  caller. This is what makes "build an image, then download it" and
  "change config â†’ rebuild" first-class rather than ad hoc.
- **Version compatibility (`requires:`).** Every plugin declares the **core
  version** it works with and the **other plugins** it depends on, each as a
  semver range â€” see the resolution model below. Plugins are semantically
  versioned (git tags); `api_version` stays the *contract-schema* version, while
  `requires.eve` is the *core feature* version.

Everything else (commands, actions, access, capabilities, supports, env,
provision steps, per-OS install trees) stays as-is â€” it is already plugin-shaped.

#### Version resolution â€” single-version, fail-on-conflict (poetry-style)

```yaml
# in any eve-plugin.yaml
requires:
  eve: ">=4.0,<5.0"          # compatible core/contract version
  plugins:
    ubuntu: ">=1.2,<2.0"     # this package needs the ubuntu provider 1.x â‰Ą1.2
    desktop-xfce: "^1.0"
```

- **Core gate:** a plugin whose `requires.eve` excludes the running eve version
  is rejected at load with a clear message â€” no silent "works mostly."
- **Dependency graph:** `eve pull` builds the graph from `requires.plugins`,
  resolves **exactly one version per plugin id** from the available pinned
  sources/tags, checks all ranges, and writes the resolved set to
  `.eve/plugins.lock`.
- **Conflicts are hard errors, like poetry/pip.** If plugin A needs `foo "^1"`
  and plugin B needs `foo ">=2"`, there is no satisfying single `foo` â€” resolution
  **fails** with a readable conflict report naming A, B, and the incompatible
  ranges. eve does **not** load two versions of one plugin to paper over it. The
  user resolves it by bumping/pinning a source, exactly as they would a Python
  dependency conflict.
- **Determinism:** `eve pull --frozen` re-materializes the locked set without
  re-resolving; `eve plugin list` shows the resolved version and which range
  required it.

#### Distribution/version constraints are per-package, not per-repo

The repo a package lives in (`eve-packages-linux`) is an **organizational
grouping**, not a compatibility claim. "Linux" does **not** mean "any Linux": a
package is typically pinned to a specific **distribution and version** (e.g.
Ubuntu 26.04) and would not work on Arch. The contract already carries this and
v4.0 keeps enforcing it â€” every package declares its precise support via
`supports`: `os_families`, `os_ids`, `os_versions`, `{family}_versions` (e.g.
`ubuntu_versions: ["26.04"]`), and `arches`. Core's `support_allowed`/
`validate_package_support` already rejects a package on a non-matching instance,
so **eve never offers an incompatible package** â€” a Windows instance is never
shown Linux packages, and an Arch instance is never shown an Ubuntu-only one.

#### Same package id across OS repos (replaces a "cross" repo)

The ~4 tools that today ship one manifest with both `install.ubuntu` and
`install.windows` trees (`rustdesk`, `nomachine`, `splashtop`, `sunshine`) are
instead defined **separately in each OS repo** â€” a Linux `rustdesk` in
`eve-packages-linux`, a Windows `rustdesk` in `eve-packages-windows`, each a
single-OS manifest with its own launcher (no in-package OS branching). Contract
rule: **the same package id may exist in multiple sources iff their `supports`
are disjoint**; eve selects the definition matching the instance's OS/distro.
Overlapping `supports` for one id is the conflict case and requires explicit
`EVE_PLUGIN_ALLOW_OVERRIDE`. This removes the need for a shared/cross repo and
keeps every repo single-OS and independently testable on its native runner.

### Source model (`.eve/plugin-sources.yaml`)

The source unit is **`(url, subdir, ref)`**, not `(url, ref)`. A single repo can
contribute **multiple independent subfolders, each pinned to its own ref** â€” so
`subdir: a` can track `main` while `subdir: b` tracks `feature-b`, from the same
repo, at the same time. This is what lets a plugin live in a subfolder *alongside
an existing project* and lets unrelated plugins co-habit one repo without being
forced onto one branch.

```yaml
sources:
  # Two independent subfolders of the SAME repo, on DIFFERENT refs:
  - id: router-stable
    url: git@github.com:fruwehq/home-router.git
    subdir: eve/openwrt
    ref: main
  - id: router-wip
    url: git@github.com:fruwehq/home-router.git
    subdir: eve/experiments
    ref: feature-b
  # A whole upstream provider, public, pinned to a tag:
  - id: provider-vultr
    url: https://github.com/fruwehq/eve-providers.git
    subdir: vultr
    ref: v1.4.0
    auth: none              # ssh (agent/keys) | token (GH_TOKEN/gh) | none
```

- **Multiple refs of one repo:** the sync engine keys its working copies by
  `(url, ref)` â€” a bare clone fetched once per repo, materialized per ref via
  `git worktree` (or per-`(url,ref)` checkout), then **sparse-checkout** of each
  `subdir`. So `home-router@main:eve/openwrt` and `home-router@feature-b:eve/experiments`
  coexist without clobbering each other.
- **Private GitHub:** default to **SSH remotes** (reuse the user's ssh-agent /
  keys â€” no secret stored by eve); `auth: token` uses `gh auth`/`GH_TOKEN` via
  git credential helper; `auth: none` for public repos over HTTPS. Never write
  tokens to disk.
- **Subfolder root:** within the chosen `subdir`, eve looks for
  `{providers,packages}/*/eve-plugin.yaml` (or a single plugin at the subdir
  root). One subdir = one independently-versioned plugin set.
- **Lockfile (`.eve/plugins.lock`):** `eve pull` resolves each
  `(url, subdir, ref)` to a concrete commit SHA and records it, so a set
  resolves reproducibly and `eve pull --frozen` re-materializes an exact state
  (CI, teammates).
- **Regular pull:** `eve pull` is manual; `eve pull --if-stale <ttl>` (and an
  optional user-installed scheduled task) covers "pull regularly."

### SSH-readiness boundary â€” the provider/package dividing line

A core invariant of eve, **already true today and made explicit here**: an
`up`'d instance is **SSH-reachable before any package provisioning runs**.
Everything *minimally necessary to reach an SSH session* â€” infra stand-up,
cloud-init / first-boot bring-up, the access/bootstrap user, the OS init
(today's `inits`: `ssh-ubuntu-cloud-init`, `ssh-windows-powershell7`) â€” is owned
by the **provider tier** and runs **pre-provisioning**. Everything *after* SSH â€”
VNC, RDP, Sunshine, desktops, dev tools, OpenWrt builds â€” is a **package** and
runs over that SSH session.

Consequences for the split:
- A **provider plugin** must deliver an SSH-ready machine on its own; it carries
  its init/bootstrap, not a separate package. Providers therefore stay minimal
  and stable (they rarely change), which is exactly why they belong in their own
  small, independently-tested repos.
- A **package plugin** may assume SSH exists and never re-implements bring-up.
- The `eve plugin test` harness asserts this boundary: a provider must reach
  SSH-ready with zero packages selected; a package must declare it requires an
  SSH-ready host and must not contribute pre-SSH steps.

### Discovery & precedence

`.eve/plugins/<source-id>/<subdir>/{providers,packages}/*/eve-plugin.yaml`.
Core already merges roots; v4.0 (a) drops `<repo>/plugins` as a default root
(core is empty), (b) makes synced plugins the norm, (c) defines deterministic
precedence when two sources define the same id (explicit `EVE_PLUGIN_ALLOW_OVERRIDE`
+ source order), and (d) lets a later source *extend* a provider (e.g. OpenWrt
packages attach to the `ubuntu` provider without forking it).

**Dogfooding â€” "eat our own dog food."** The first-party plugins are not special:
each provider subfolder and each package starts as a **copy of
`eve-plugin-template`**, then has its logic ported in. If the template can't
express a built-in, the template (and likely the contract) is wrong â€” so building
the core content from the template is the contract's primary acceptance test.

**Why this split (test-suite size):** providers are minimal, stable, and rarely
change (they own only "stand up â†’ SSH-ready"), so isolating them keeps their CI
tiny and out of the hot path. Splitting packages by OS lets `eve-packages-linux`
CI on a Linux runner and `eve-packages-windows` on a Windows runner, each testing
only its own provisioning â€” instead of one suite exercising both. Each provider
subfolder and each package repo runs `eve plugin test` independently against the
pinned contract, with **no `eve` checkout**. The cross-platform tools are defined
separately per OS repo (see the "same package id across OS repos" rule above) â€”
**there is no `-cross` repo**.

> **Org namespace:** the `fruwehq` GitHub org (bare `fruwe` is the personal
> account). An org gives ownership continuity, per-repo visibility/permissions
> (public `eve`/template/providers mixed with private user content), and free
> transfers/redirects.

> **Granularity is a source-list change, not architecture.** Starting with
> `eve-packages-linux`/`-windows` and later splitting (e.g.
> `eve-packages-streaming`) â€” or adding an `eve-packages-arch` for a new distro â€”
> is just editing `plugin-sources.yaml`.

### Repository migration (this repo â†’ the org's `eve`)

**Transfer and rename this repo; do not start `eve` fresh.** The core *is* this
repo minus the plugins â€” the agnostic-core work (Phases 0â€“3) removes providers/
packages, it does not rebuild the SDK/dispatch/schema/catalog/CLI, which evolved
here. So:

- **GitHub-transfer `fruwe/ephemeral-cloud-gaming` into the org and rename it to
  `eve`.** This preserves git history, issues, merged PRs, and CI config, and
  sets up redirects from the old owner/name so existing clones and links keep
  working. A fresh repo would re-import the same code with none of that.
- **Secrets:** repo-scoped Actions secrets do **not** follow a transfer. There are
  **none today**, so no action now â€” but when cloud credentials (`VULTR_API_KEY`,
  etc.) are added later, prefer **org-level** secrets so all plugin repos share
  them.
- **Settle open PRs first** (e.g. close the stale pre-v3 agent-orchestration PR,
  whose external-config-repo idea is superseded by this registry) so nothing
  in-flight has to be untangled across the move.
- **Plugin history at extraction (Phase 3):** per repo, either `git filter-repo`
  the subtree to keep blame (worth it for `plugins/providers/` â€” long-lived,
  debugged code) or start fresh from `eve-plugin-template` (fine for packages,
  which are scaffolded from the template anyway and have their provisioning logic
  ported in). Core keeps its history in place; plugin repos keep theirs only
  where blame is valuable.

Sequence: settle PRs â†’ transfer to the org â†’ rename to `eve` â†’ (re-add secrets if/
when they exist) â†’ run the v4.0 phases, which extract plugins outward.

### How the user's examples map onto the contract

- **Providers as plugins with ssh.** A `ubuntu`/`windows` provider plugin already
  exposes `ssh`, `up/down/start/stop/status/ip` commands and `ssh` as an action
  target â€” exactly "providers, as the plugins they are, provide ssh actions." No
  new concept; just relocated out of core.
- **OpenWrt router.** `eve-plugins-openwrt` ships packages whose `supports`
  targets the `ubuntu` provider. Provision steps install the OpenWrt build
  toolchain and apply config from the package's `config_schema`
  (`openwrt.wifi_ssid`, `openwrt.profile`, â€¦). A `build` action runs the image
  build on the instance; a `download-image` action (artifact convention) scp's
  the result back. Editing `.eve/config.yaml` and re-running
  `eve package provision openwrt` (or an explicit `rebuild` action) re-applies
  config and rebuilds â€” the artifacts + `config_schema` additions exist to make
  this loop first-class.
- **AI workstation.** `eve-plugins-ai` contributes a `bundle` (e.g.
  `ai-desktop` = `xfce-desktop` + `vscode` + `claude` + â€¦) on Ubuntu. Bundle
  contribution is the package-manifest `bundles:` addition.

### CLI (gcloud-style tree) and performance

A single Python entrypoint with argparse subparser groups replaces the Make
surface:

```
eve instance  create|list|up|down|start|stop|status|ssh|ip|provision|reboot|logs
eve package   list|add|install|status|down|action|provision
eve provider  list|status
eve bundle    list|select
eve plugin    source add|list|remove | test            # conformance harness
eve pull      [--frozen|--if-stale <ttl>]
eve config    get|set|list
eve tui
eve test      â€¦                                         # dev
```

Make is retained only as thin dev aliases (or removed). The same entrypoint
backs a **warm session**: catalog + plugins are parsed once per process and the
TUI / a `eve batch -f pipeline.yaml` (or stdin script) run many operations
without re-cold-starting â€” the in-process API the TUI calls instead of shelling
out to `scripts/*` per action. This extends v3.5's amortization to the
interactive surface.

## Phased delivery

Each phase is independently shippable and leaves the full test suite green.
Phases 0â€“5 land in `fruwehq/eve` (Phase 0 also publishes `eve-plugin-template`);

**Predecessor (near-term, already scoped):** [v3.5](v3.5-roadmap.md) â€” sub-15-min

## Non-goals

- Rewriting guest-side provisioning logic (the bash/PowerShell that runs *on*
  instances stays; only its *packaging* moves to external repos).
- Building a hosted plugin marketplace/registry service â€” sources are plain git
  endpoints; discovery is the user's source list, not a central index.
