# v4.3 Roadmap — Cleanup & fixes to finalize v4.x

Status: in progress (2026-06-18). v4.0–v4.2 built the agnostic plugin host
(v4.0), the lean/decoupled test surface + adoption story (v4.1/v4.2), and
plugin-source management (v4.2). v4.3 is the **polish pass that finalizes
milestone 4**: make `eve` the single, fast, coherent entrypoint, fix the UX
rough edges the host surfaced once it was actually used, and remove the
redundant `make` interface the TUI still leans on.

## Philosophy — one entrypoint, no redundant layers

`eve` is the interface. The CLI and TUI are two faces of the same in-process
Engine; `make` is for development/CI only, not an operational API. v4.3 removes
the leftover seams (TUI → `make`, TUI as the default, cryptic empty-state
errors) so a first run is fast and legible.

## Goals

1. `eve` is fast and help-first; the TUI is explicit.
2. No operational `make` layer — the TUI drives the same scripts/Engine the CLI
   does; the Makefile is dev/test/build only.
3. Empty-state (no providers/plugins) is legible everywhere — no cryptic errors,
   no offering actions that can't work.
4. The TUI is regression-covered (headless), not verified by hand each time.

## 1. CLI & startup (DONE)

- `eve` with no args prints **help**; the TUI is explicit (`eve tui`).
- Groups + verbs listed **alphabetically**.
- **Startup ~0.70s → ~0.09s**: `bin/eve` execs the in-project `.venv` python
  directly instead of `poetry run` (falls back to `poetry run`).
- Actionable error when **no plugins are installed** (was the cryptic
  `Machine not found: local-qemu-medium`).

## 2. TUI UX (DONE)

- **Lifecycle gating**: hide/disable up/stop/down/provision/recover when no
  providers are installed; local actions (refresh, delete entry, settings,
  plugins) stay.
- **Plugin screen**: "Add URL…" opens a modal; explicit "Add selected" (+ `a`)
  adds a recommended source.
- **Refresh** moved to a small non-primary `⟳` button by the "Instances" title;
  Enter/Space on a row "goes inside" (focuses the right pane).
- **Headless TUI smoke tests** (Textual `run_test`) — real coverage the
  pure-helper tests can't give.

## 3. Remove the operational `make` layer (TODO — the main v4.3 work)

The TUI still shells out to `make` for every mutation (`make_args` /
`package_make_args` / `bundle_make_args`, `make providers.status`, `make ip`),
which is why ~85 make targets persist. The CLI already calls `scripts/` directly
(via `build_command`); the reads already use the warm Engine. Finish the job:

- Repoint the TUI's mutations off `make` onto the underlying scripts (or the
  Engine's op methods), as pure, unit-tested arg builders. Full target→script
  map is known.
- **Gotcha:** the Makefile injects config-derived env (`config-env --make` →
  `AWS_PROFILE`, `TIMEZONE`, …) before each recipe. Verify scripts load config
  standalone (the CLI calls them the same way, so they should) via a mock-fixture
  mutation in the headless harness before pruning.
- Prune the Makefile to **dev/test/build only** (`test` + `test.*` — CI needs
  these, incl. the macOS `test.doctor test.python` step — `generate`, `clean`,
  `lint`, `install-cli`, `eve`, `tui`). Remove operational targets.

## 4. Finalization odds & ends

- Per-repo READMEs for `eve-providers`, `eve-packages-linux`,
  `eve-packages-windows` (eve + eve-plugins-ai done in v4.2).
- Optional: `space` already added to list activation; revisit any remaining
  empty-state legibility (e.g. TUI hint when no sources configured).

## Non-goals

- New features — v4.3 is cleanup/finalization only.
- Re-introducing a second operational interface (make stays dev-only).

## Sequencing

1. §1/§2 CLI + TUI fixes (done; first PR).
2. §3 remove the `make` layer (TUI → scripts/Engine, then prune Makefile).
3. §4 docs/finalization.
