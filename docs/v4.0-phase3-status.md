## Parked for review (the risky parts I did NOT do blind)

### A. Core integration — empty `plugins/` + load from the external repos
This is the agnostic-core milestone, but its **blast radius is large**: ~29 test
files reference builtin plugin ids, and `test-plugins` / `test-instances` /
`test-lifecycle` + all the goldens assume builtins in `plugins/`. Emptying core
requires a **suite-wide loading rework** (have the suite `eve pull` / set
`EVE_PLUGIN_ROOTS` to the external repos) and a committed default-sources strategy.
Decisions needed:
1. **How the suite/runtime gets the plugins by default.** Cleanest is a committed
   `plugin-sources.yaml` pinned at `fruwehq/eve-{providers,packages-linux}` +
   `eve pull` in test setup — which means **pushing the plugin repos** (sanctioned
   "inter-repo dependency" push; cheap conformance CI only, not the expensive eve
   suite). Alternative for pure-local: `EVE_PLUGIN_ROOTS` at the local clones
   (works locally, not a committable default).
2. **Golden-parity test** must move from "read builtins in `plugins/`" to
   "`eve pull` → `catalog-options` → compare golden". Expected byte-identical
   (the manifests are verbatim copies), but it touches the test harness.

### B. Dual-OS split + disjoint-supports core change (deferred)
The 8 dual-OS tools should become **single-OS** manifests: a `[ubuntu]` variant in
`eve-packages-linux` and a `[windows]` variant in `eve-packages-windows` (same id,
disjoint `supports`). That requires a **core change**: `load_all` must let the same
package id coexist across sources when `supports` are disjoint, and the catalog
aggregator / `catalog-options` must **merge** them back so output stays
byte-identical to the golden. This is the one genuinely parity-risky core change —
deferred so it can be done with you available and the integration (A) working as
the test substrate. For now the whole manifests reproduce today's behavior exactly.
